- ORM(Object-Relational Mapping)은 객체(Object)와 관계형 데이터베이스(Relational Database)의 데이터를 자동으로 연결(Mapping)해주는 기술

- SQL 문을 직접 작성하지 않고도 자바나 파이썬 같은 **프로그래밍 언어의 객체**를 통해 데이터베이스를 다룰 수 있게 해주는 도구

---
- **기존 방식 (SQL 중심):** 개발자가 직접 `SELECT * FROM users WHERE id = 1` 같은 쿼리를 작성하여 DB에 전달, 결과로 받은 데이터를 다시 객체 형태로 변환하는 번거로운 작업 필요

- **ORM 방식 (객체 중심):** `User.find(1)`과 같은 코드를 실행하면, ORM이 내부적으로 적절한 SQL을 생성하여 실행하고 결과를 객체에 담아 반환

---
# ORM의 장단점

### 장점

- **생산성 향상:** 반복적인 SQL 작성 대신 선언적인 코드를 사용하므로 비즈니스 로직에 더 집중할 수 있습니다.

- **유지보수 용이:** 테이블의 컬럼이 변경되더라도 SQL 문을 일일이 찾아 수정할 필요 없이, 객체 모델만 수정하면 됩니다.

- **DBMS 독립성:** ORM 설정만 바꾸면 MySQL에서 PostgreSQL로 데이터베이스를 교체해도 코드를 거의 수정하지 않아도 됩니다. (ORM이 각 DB에 맞는 SQL을 대신 생성해주기 때문입니다.)

- **가독성:** SQL보다 객체 지향적인 코드가 가독성이 높고 코드 양이 줄어듭니다.

### 단점

- **성능 저하 위험:** ORM이 자동으로 생성하는 SQL이 최적화되지 않을 수 있습니다. 특히 복잡한 조인(Join)이 필요한 경우 성능이 크게 떨어질 수 있습니다.

- **N+1 문제:** 연관된 데이터를 조회할 때 1번의 쿼리로 끝날 작업이 N번 더 실행되는 대표적인 성능 저하 문제입니다.

- **학습 곡선:** 단순한 조회는 쉽지만, 성능 최적화나 복잡한 쿼리를 위해 ORM의 내부 동작 원리를 깊게 공부해야 합니다.

> ORM은 객체 지향적인 코드를 통해 생산성을 높여주지만, 잘못 사용하면 불필요한 쿼리가 발생하여 성능에 악영향을 줄 수 있음. 따라서 **프로젝트의 규모와 성능 요구사항에 따라 ORM을 적절히 사용**하되, 복잡한 쿼리는 직접 SQL을 작성하거나 QueryDSL 같은 도구를 병행하여 최적화해야 함

# 대표적인 ORM 도구

- **Python:** **SQLAlchemy** (FastAPI에서 주로 사용), **Django ORM**
- **Java:** **Hibernate** (JPA의 구현체)
- **JavaScript/Node.js:** **Sequelize**, **Prisma**
