- 트랜잭션(Transaction)은 데이터베이스 관리 시스템(DBMS)에서 **하나의 논리적 기능을 수행하기 위한 작업의 단위**

- 여러 개의 작업을 하나로 묶어서 "모두 성공하거나, 아니면 아예 아무것도 안 한 상태로 되돌리거나"를 보장하는 것

---
# 트랜잭션의 4가지 특징 (ACID)

- **원자성 (Atomicity):** "All or Nothing". 트랜잭션 내의 모든 명령은 반드시 완벽히 수행되어야 하며, 어느 하나라도 오류가 발생하면 트랜잭션 전부가 취소(Rollback)되어야 합니다.

- **일관성 (Consistency):** 트랜잭션이 완료된 후에도 데이터베이스는 미리 정의된 규칙(제약 조건)을 만족해야 합니다. (예: 잔액은 0보다 커야 함)

- **격리성 (Isolation):** 둘 이상의 트랜잭션이 동시에 실행될 때, 서로의 연산에 끼어들 수 없습니다. 수행 중인 트랜잭션은 완료될 때까지 다른 트랜잭션에서 참조할 수 없습니다.

- **영속성 (Durability):** 성공적으로 완료된 트랜잭션의 결과는 시스템에 장애가 발생하더라도 영구적으로 기록되어야 합니다.

---
# 트랜잭션의 상태

- **Active (활동):** 트랜잭션이 실행 중인 상태.

- **Partially Committed (부분 완료):** 마지막 명령까지 실행했지만, 아직 최종 결과를 반영하기 전 상태.

- **Committed (완료):** 트랜잭션이 성공적으로 종료되어 변경 내용을 DB에 영구 저장한 상태.

- **Failed (실패):** 오류가 발생하여 실행이 중단된 상태.

- **Aborted (철회):** 트랜잭션을 취소하고 실행 전 상태로 되돌린(Rollback) 상태.

---
# 트랜잭션 연산

- **Commit 연산**
	- 변경 사항을 물리적 디스크에 기록
	- 트랜잭션이 **'부분 완료(Partially Committed)'** 상태에서 **'완료(Committed)'** 상태로 넘어가게 만드는 행위

- **Rollback 연산**
	- 메모리상에만 존재하던 변경사항 폐기
	- 트랜잭션이 **'활동(Active)'** 중 오류가 나거나, **'부분 완료'** 상태에서 일관성을 위배했을 때 '실패(Failed)'를 거쳐 **'철회(Aborted)'** 상태로 가게 만드는 행위

> DBMS나 프레임워크에는 오토 커밋(Auto-commit) 모드가 설정되어 있어 SQL 문 하나하나를 각각의 독립된 트랜잭션으로 간주하고, 명령어가 성공적으로 실행되면 DBMS가 자동으로 COMMIT을 호출함

---
# 트랜잭션 격리수준

- 여러 트랜잭션이 동시에 처리될 때, 특정 트랜잭션이 다른 트랜잭션에서 변경하거나 조회하는 데이터를 볼 수 있게 허용할지 말지를 결정하는 설정

- 아래로 갈수록 격리성이 높아지며 데이터 부정합 문제는 줄어들지만, 동시 처리 성능은 떨어짐

|**격리 수준**|**발생 가능한 문제**|**특징**|
|---|---|---|
|**Read Uncommitted**|Dirty Read, Non-Repeatable Read, Phantom Read|커밋되지 않은 데이터도 읽음 (성능은 극대화되나 위험함)|
|**Read Committed**|Non-Repeatable Read, Phantom Read|커밋된 데이터만 읽음 (대부분의 DB 기본값 또는 권장값)|
|**Repeatable Read**|Phantom Read (일부 DB 제외)|트랜잭션 내에서 한 번 읽은 데이터는 반복해서 읽어도 동일함|
|**Serializable**|없음|트랜잭션을 순차적으로 실행함 (가장 안전하지만 가장 느림)|
- MySQL(InnoDB)은 **Repeatable Read**를, Oracle이나 PostgreSQL은 **Read Committed**를 기본으로 사용
