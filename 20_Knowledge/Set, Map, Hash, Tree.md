# Set vs Map

- **Set(집합):** **값(Value)**만 저장, 중복 허용X, 순서 보장X
- **Map(지도):** **키(Key)**와 **값(Value)**의 쌍으로 저장, 키는 중복X, 값은 중복O

---
# Hash vs Tree
### Hash

- **원리:** `HashCode()`를 이용해 데이터의 위치(인덱스)를 즉시 계산
- **성능:** 추가, 삭제, 검색이 평균적으로 **O(1)**으로 매우 빠르다
- **특징:** 데이터가 들어온 순서를 알 수 없으며, 정렬되지 않음
### Tree

- **원리:** 이진 탐색 트리(Binary Search Tree), 그 중에서도 균형을 잡는 Red-Black Tree 구조 사용
- **성능:** 추가, 삭제, 검색에 **O(logN)**의 시간 소요
- **특징:** 데이터가 **정렬된 상태**로 유지(기본은 오름차순)

# 정리
| **분류**  | **구현체**     | **특징**               | **시간 복잡도**  | **비고**                |
| ------- | ----------- | -------------------- | ----------- | --------------------- |
| **Set** | **HashSet** | 중복 불가, 순서 없음         | $O(1)$      |                       |
|         | **TreeSet** | 중복 불가, **정렬됨**       | $O(\log N)$ | 값에 따라 정렬이 필요할 때 사용    |
| **Map** | **HashMap** | 키 중복 불가, 순서 없음       | $O(1)$      |                       |
|         | **TreeMap** | 키 중복 불가, **키 기준 정렬** | $O(\log N)$ | 키를 기준으로 정렬/범위 검색 시 사용 |
- **HashSet / HashMap을 선택하는 경우**
	- 단순히 **데이터가 있는지 없는지** 확인하거나, **키로 값을 빠르게 찾기**만 하면 될 때
	- 특별한 이유가 없다면 성능이 가장 좋은 Hash 계열 사용

- **TreeSet / TreeMap을 선택하는 경우**
	- 데이터가 **정렬된 상태**를 유지해야 할 때
	- 정렬과 범위 탐색이 핵심 요구사항일 때 사용
