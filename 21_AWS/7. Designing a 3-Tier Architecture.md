# 아키텍처 개선

### 기존 아키텍처

![[../80_Assets/Pasted image 20260719153511.png]]

- 기능을 검증하고 사용자 피드백을 수집할 수 있는 간단하며 비용 효율성이 우수한 애플리케이션부터 구축하고자 함
- 이후 애플리케이션 완성도가 높아지고 수요가 증가하면, 높은 가용성과 성능을 유지하고 대규모로 확장할 수 있도록 **로드 밸런싱과 오토 스케일링, 향상된 모니터링** 방식 등의 다양한 기능을 아키텍처에 추가
- 단일 AWS 가용 영역에 개념 증명(PoC) 배포용으로 구축한 기본적인 3티어 웹 애플리케이션 아키텍처
	- 사용자는 인터넷 게이트웨이를 통해 퍼블릭 서브넷의 시스템에 액세스
	- 해당 퍼블릭 서브넷에는 Amazon EC2 웹 서버와 NAT 게이트웨이가 있음
	- 트래픽은 EC2 앱 서버가 있는 프라이빗 앱 서브넷으로 전송
	- 이 EC2 앱 서버는 데이터베이스 서브넷의 Amazon RDS 인스턴스에 연결

### 개선 아키텍처

![[../80_Assets/Pasted image 20260724001643.png]]

- AWS 리전 1개에 **가용 영역 2개**
- 사용자는 **인터넷 게이트웨이 및 외부 로드 밸런서**를 통해 앱에 액세스
- 트래픽은 **퍼블릭 서브넷**의 **Auto Scaling** EC2 웹 서버로 라우팅
- 내부 **로드 밸런서**는 **프라이빗 서브넷**의 EC2 앱 서버로 요청을 전달, 이러한 앱 서버도 **Auto Scaling 그룹**에 포함
- 앱 서버는 데이터베이스 서브넷 1개의 **Amazon RDS** 프라이머리 인스턴스와 다른 서브넷의 **읽기 전용 복제본**에 연결
- **NAT 게이트웨이**를 통해 프라이빗 인스턴스로의 아웃바운드 인터넷 액세스가 지원

# 모니터링

![[../80_Assets/Pasted image 20260721192810.png]]

- 리소스가 운영되고 수행되는 상황을 추적하려면 모니터링이 필요
	- 리소스 사용률 및 애플리케이션 성능에 대한 정보를 수집
	- 인프라가 충분한지 여부를 측정
	- 인프라를 스케일 아웃 또는 스케일 인해야 하는 시기를 확인
	- 보안 측면에서 사용자가 환경의 일부에 액세스 여부 및 시기를 파악하고 권한을 확인

### Amazon CloudWatch

![[../80_Assets/Pasted image 20260721193611.png]]

- **Amazon CloudWatch**는 시스템 이벤트 스트림을 거의 실시간으로 제공하는 AWS 서비스
- CloudWatch 한곳에서 모든 AWS 리소스, 애플리케이션, 서비스에 걸쳐 이 데이터를 수집 및 액세스하고 상관관계를 파악할 수 있음
- CloudWatch는 온프레미스 서버에서도 데이터를 수집할 수 있음
- 성능 및 리소스 사용을 최적화할 수 있도록 CloudWatch에서는 자동 대시보드를 제공하며, 1초 세부 단위로 데이터를 제공하고 최대 15개월 동안 지표를 저장 및 보존

- **CloudWatch 지표**는 시스템 성능에 대한 데이터
- 대다수 서비스에서는 기본적으로 리소스 관련 지표를 제공 (EC2, EBS, RDS 등)
- EC2 인스턴스 같은 일부 리소스에 대한 세부 모니터링을 활성화하거나 자체 애플리케이션 지표를 게시할 수 있음
- CloudWatch는 검색, 그래프 처리 및 경보를 위해 계정에 모든 지표(AWS 리소스 지표 및 사용자가 제공한 애플리케이션 지표 모두)를 로드할 수 있음

### AWS CloudTrail

- **AWS CloudTrail**은 사용자 활동 및 API 사용량을 추적하여 누가 무엇을 언제 했는가에 대한 인사이트를 제공
- CloudTrail에서는 콘솔, AWS SDK, AWS CLI, 상위 수준 AWS 서비스를 통해 이루어진 호출을 포함하여 계정에 대한 AWS API 직접 호출 내역을 확인할 수 있음
- CloudTrail은 AWS API 직접 호출을 기록하고 로그 파일을 제공
- 이 정보에는 API 직접 호출자의 소스 IP 주소 및 자격 증명, API 직접 호출 시간, 요청 파라미터 및 AWS 서비스가 반환한 응답 요소가 포함
- 이를 활용하여 보안 분석, 리소스 변경 사항 추적 및 규정 준수 감사를 수행할 수 있음
	- 인스턴스가 종료된 이유가 무엇이며, 누가 종료했는가? (조직적 추적 기능 및 책임)
	- 누가 보안 그룹 구성을 변경했는가? (책임 및 보안 감사)
	- 권한 부족으로 인해 거부된 활동은 무엇인가? (네트워크에 대한 잠재적 내외부 공격)

- CloudTrail은 리전 단위로 활성화되며, 여러 리전을 사용하는 경우 리전별로 로그 파일이 전송될 장소를 선택할 수 있음. CloudTrail은 로그를 지정된 S3 버킷에 저장

| 구분           | AWS CloudTrail                    | Amazon CloudWatch                  |
| ------------ | --------------------------------- | ---------------------------------- |
| **주요 목적**    | 보안 감사, 거버넌스, 규정 준수 및 계정 활동 추적     | 리소스/애플리케이션 모니터링, 알람 설정 및 성능 최적화    |
| **핵심 질문**    | "이 EC2 인스턴스를 종료한 사람은 누구인가?"       | "이 EC2 인스턴스의 CPU 사용량이 90%를 초과했는가?" |
| **수집 데이터**   | 계정 활동, API 호출 기록, 로그인 시도 등        | 로그 파일, 메트릭(지표), 이벤트 스트림 등          |
| **주요 활용 사례** | 보안 침해 사고 조사, 규정 준수(Compliance) 보고 | 시스템 장애 감지, 자동화된 리소스 대응 조치          |
- 두 서비스는 상호 보완적으로 사용되며, 일반적으로 CloudTrail에서 수집한 보안 감사 로그를 CloudWatch Logs로 전송하여 비정상적인 API 활동이 발생할 때 실시간으로 경고를 받도록 구성하는 경우가 많음

### VPC Flow Logs

- VPC 흐름 로그는 VPC 네트워크 인터페이스에서 송수신되는 IP 트래픽 정보를 캡처
	- 이러한 로그는 VPC, 서브넷 또는 네트워크 인터페이스 기준으로 트래픽을 기록하도록 구성할 수 있음
	- EC2 및 VPC 콘솔에서 특정 리소스의 흐름 로그 탭을 선택하여 흐름 로그에 대한 정보를 볼 수 있음
	- 흐름 로그는 기본적으로 해제되어 있음. 흐름 로그 데이터를 수집하려면 옵트인해야 함

- 각 레코드는 특정 캡처 기간에 대한 네트워크 IP 흐름을 캡처하며 5가지 값(**5-tuple**)을 포함
	- 레코드에는 소스, 대상, 프로토콜 같은 IP 흐름의 구성 요소가 포함
	- 특정 유형의 트래픽이 탐지되면 활성화되는 경보를 생성하고 경향 및 패턴을 식별하는 데 도움이 되는 지표를 생성할 수 있음

### CloudWatch 경보

- CloudWatch 경보는 **지표 경보**라고 하는 단일 클라우드 지표 또는 **복합 경보**라고 하는 여러 클라우드 지표의 구성을 기반으로 생성할 수 있음
- 경보는 여러 기간 동안의 임곗값과 비교하여 지표 또는 표현식의 값을 기준으로 1개 이상의 작업을 수행
- 경보에는 3가지 상태가 있음
	- **OK** - 지표가 정의된 임곗값 내에 있음
	- **ALARM** - 모니터링되는 지표가 지정된 임곗값과 같거나, 그보다 크거나 작다는 의미
	  (e.g. 지정된 EC2 인스턴스의 CPU 사용률이 너무 높을 때를 알려 주는 경보)
	- **INSUFFICIENT_DATA** - 지정된 지표에 대한 데이터가 없을 경우

![[../80_Assets/Pasted image 20260721222837.png]]
- 위 예제에서는 경보 임곗값이 25%로 설정되어 있고 최소 위반은 2개 기간
- 즉, 임곗값이 2개 기간 연속으로 위반된 경우에만 경보 상태가 변경

### Amazon EventBridge

![[../80_Assets/Pasted image 20260723164808.png]]

- **Amazon EventBridge**는 애플리케이션, 통합 SaaS 애플리케이션 및 AWS 서비스에서 생성된 이벤트를 사용하여 대규모 이벤트 기반 애플리케이션을 쉽게 구축할 수 있도록 하는 서버리스 이벤트 버스
- CloudWatch에서 캡처된 이벤트를 관리할 때는 기본적으로 EventBridge를 사용

# 로드 밸런싱

- 웹 애플리케이션은 트래픽이 가장 많은 기간에도 그리고 트래픽이 적은 기간에도 효율적으로 실행되어야 함

### Elastic Load Balancing (ELB)

![[../80_Assets/Pasted image 20260721234014.png]]

- ELB 로드 밸런서는 네이티브 방식으로 사용자를 EC2 인스턴스, 컨테이너 배포 및 AWS Lambda 함수에 연결하는 AWS에서 제공하는 유일한 로드 밸런서
- 로드 밸런서는 리전 범위 수준 서비스

- **고가용성** - ELB는 단일 가용 영역 또는 여러 가용 영역에서 다수의 대상에 걸쳐 트래픽을 자동으로 분산
- **계층 4 또는 계층 7 HTTP 및 HTTPS 로드 밸런싱** - 계층 7 전용 기능에 대해 HTTP 또는 HTTPS 애플리케이션을 로드 밸런싱 하거나, TCP만 사용하는 애플리케이션에 대해 엄격한 계층 4 로드 밸런싱을 사용할 수 있음
- **보안 기능** - VPC를 사용하면 로드 밸런서와 연결된 보안 그룹을 생성 및 관리하여 추가 네트워킹 및 보안 옵션을 제공할 수 있음. 또한 내부(인터넷에 연결되지 않음) 로드 밸런서를 생성할 수도 있음
- **상태 확인** - ELB 로드 밸런서는 비정상 대상을 감지하고, 해당 대상으로 트래픽 전송을 중단한 다음, 나머지 정상 대상으로 로드를 분산
- **모니터링 작업** - ELB는 애플리케이션 성능을 실시간으로 모니터링할 수 있도록 CloudWatch 지표와 통합되며 요청 추적 기능을 제공

### 로드 밸런서 유형

- **OSI 7계층** - **P**hysical / **D**ata / **N**etwork / **T**rasport / **S**ession / **P**resentation / **A**pplication
	- **P**lease **D**o **N**ot **T**rust **S**ales-**P**erson's **A**dvice (영업사원의 조언을 믿지마라)
	
- 로드 밸런서의 계층은 패킷을 어느 수준까지 읽어서 판별할 수 있는지 결정
	- 인터넷 패킷은 겹겹이 포장된 택배 상자
	- 각 로드밸런서는 자기가 볼 수 있는 깊이까지만 뜯어보고 목적지를 결정
```
[3계층 헤더: IP 주소] -> [4계층 헤더: 포트 번호] -> [7계층 데이터: HTTP/URL/자체 규격]
```

- **(계층 7) Application Load Balancer**
	- HTTP 및 HTTPS 트래픽의 고급 로드 밸런싱에 적합
	- 콘텐츠 기반 라우팅, 컨테이너에서 실행되는 애플리케이션, 개방형 표준 프로토콜(WebSocket 및 HTTP/2)을 지원

- **(계층 4) Network Load Balancer**
	- TCP 및 UDP 트래픽 밸런싱에 적합
	- 높은 처리량과 짧은 지연 시간을 유지하면서 초당 수천만 건의 요청을 처리하도록 설계
	- 연결 수준(계층 4)에서 작동하여 IP 프로토콜 데이터에 따라 대상으로 연결을 라우팅

- **(계층 3) Gateway Load Balancer**
	- 서드 파티 가상 어플라이언스를 배포, 규모 조정, 관리할 수 있음
	- 여러 가상 어플라이언스에 트래픽을 분산하면서 수요에 따라 스케일 업 또는 스케일 다운하는 1개의 게이트웨이 역할
	- 서드 파티 가상 어플라이언스를 통한 모든 3 계층 트래픽을 투명하게 전달. 소스와 대상 위치에서는 이를 볼 수 없음

| 구분          | (계층 7) Application Load Balancer             | (계층 4) Network Load Balancer                    | (계층 3) Gateway Load Balancer      |
| ----------- | -------------------------------------------- | ----------------------------------------------- | --------------------------------- |
| **작동 계층**   | **7계층 (Application)**                        | **4계층 (Transport)**                             | **3계층 (Network)**                 |
| **판단 기준**   | HTTP/HTTPS 내부 데이터  <br>(URL 경로, 쿠키, HTTP 헤더) | TCP/UDP 프로토콜 정보  <br>(출발지/목적지 IP, 포트 번호)        | IP 패킷 전체  <br>(데이터 수정 없이 통째로 중계)  |
| **이해하는 언어** | 표준 **HTTP / HTTPS**만 이해 가능                   | 언어 불문 (**자체 패킷 규격**, 게임, IoT 등 모든 TCP/UDP)      | 언어 불문 (모든 IP 트래픽)                 |
| **처리 속도**   | 상대적으로 느림 (밀리초 단위)  <br>※ 패킷을 끝까지 열어봐야 하므로    | **압도적으로 빠름** (마이크로초 단위)  <br>※ 겉 주소만 보고 바로 던지므로 | 매우 빠름 (투명한 패킷 전달)                 |
| **최적화 서비스** | MSA 구조, 일반 웹/앱 서비스, URL 기반 라우팅 필요 시          | 실시간 게임, 금융 거래, 대규모 트래픽 초고속 분산 시                 | 서드파티 가상 방화벽, IPS/IDS 등 보안 장비 연동 시 |
- e.g. 기능별로 쪼개진 **마이크로서비스(MSA)** 형태의 경우, 요청 내용(URL)에 따라 라우팅 필요
  → **ALB 사용**

# 오토 스케일링

- 웹 애플리케이션의 아키텍처는 계속해서 늘어나는 고객 수요를 충족할 수 있는 확장성도 추가로 지원해야 함
- 클라우드 인프라의 한 가지 장점은 애플리케이션 요구 사항에 따라 원활하게 스케일 아웃 및 스케일 인할 수 있다는 것

### AWS Auto Scaling

- **AWS Auto Scaling**은 애플리케이션을 모니터링하고 용량을 자동으로 조정하여, 최대한 저렴한 비용으로 안정적이고 예측 가능한 성능을 유지
- AWS Auto Scaling을 사용하면 몇 분 만에 여러 서비스 전체에서 여러 리소스에 대한 애플리케이션 스케일링을 설정할 수 있음
- 스케일링 정책을 기반으로 애플리케이션 수요의 증감에 따라 인스턴스를 시작하거나 종료

![[../80_Assets/Pasted image 20260722021911.png]]

- EC2 Auto Scaling은 ELB와 통합 가능해, 1개 이상의 로드 밸런서를 EC2 Auto Scaling 그룹에 연결할 수 있음

- EC2 Auto Scaling을 구현하려면 각 인스턴스의 구성을 정의하는 시작 템플릿, Auto Scaling 그룹, 스케일링 정책이라는 3가지 구성 요소가 필요

1. **시작 템플릿**
   ![[../80_Assets/Pasted image 20260722022645.png]]
	- Amazon Machine Image(AMI) ID와 인스턴스 유형 등 EC2 인스턴스를 시작하는 데 필요한 파라미터가 포함 (어떤 EC2 인스턴스를 만들건지)
	
2. **Auto Scaling 그룹**
   ![[../80_Assets/Pasted image 20260722022703.png]]
	- 오토 스케일링 및 관리를 위한 논리적 그룹으로 취급되는 EC2 인스턴스의 모음
	- 사용자가 정의한 스케일링 정책에 따라 인스턴스 수가 최소 및 최대 인스턴스 수 내에서 지정하는 조건에 따라 조절
	
3. **스케일링 정책**
	- 스케일링 정책에 따라 스케일링 간접 호출 시기가 정의
	- 다음과 같은 도구를 사용하여 그룹에서 스케일링을 간접적으로 호출할 수 있음
		- **상태 확인** – 지정된 수의 실행 인스턴스를 항상 유지하도록 그룹을 구성. 인스턴스가 비정상 상태가 되면 그룹에서는 비정상 인스턴스를 종료하고 이를 대체할 다른 인스턴스를 시작
		- **CloudWatch 경보** – 스케일링 정책이 EC2 Auto Scaling에 특정 CloudWatch 지표를 추적하도록 설정. 이 정책은 연결된 CloudWatch 경보가 ALARM 상태일 때 수행할 작업을 정의
		- **일정** – 일정에 따라 규모를 조정. 일정에 따른 스케일링은 그룹의 인스턴스 수를 늘리거나 줄여야 할 때를 정확히 파악하고 있는 경우에 유용
		- **수동 스케일링** - 리소스 규모를 조정하는 가장 기본적인 방법. 그룹의 최대 용량, 최소 용량 또는 원하는 용량의 변경 사항만 지정. EC2 Auto Scaling은 인스턴스를 생성 또는 종료하는 프로세스를 관리하여 업데이트된 용량을 유지

- Amazon EC2에 대한 오토 스케일링은 다음 3가지 방법 중 하나를 사용하여 구현할 수 있음
  ![[../80_Assets/Pasted image 20260722023512.png]]
	- **예약 스케일링** - 웹 애플리케이션의 예측 가능한 트래픽 패턴에 따라 스케일링 활동을 계획
	- **동적 스케일링** - 수요 변화에 대응하여 Amazon EC2 Auto Scaling 그룹의 용량 규모를 조정
	- **예측 스케일링** - 트래픽 흐름의 일별 및 주별 패턴에 따라 그룹의 EC2 인스턴스 수를 조정

# 복원력

- 웹 애플리케이션의 재해 계획, 보호 및 백업을 위한 옵션
- 인스턴스 / 가용영역 / 리전 수준 장애 조치를 계획

### 재해 복구(DR) 계획

![[../80_Assets/Pasted image 20260723225921.png]]

- **RPO (목표 복구 시점, Recovery Point Objective)**
	- 장애 발생 시 **"데이터를 최대 어디까지 잃어도 되는지"** (데이터 손실 허용 범위)
	- RPO가 4시간이라면, 장애 발생 시점으로부터 4시간 전까지 백업된 데이터로 복구

- **RTO (목표 복구 시간, Recovery Time Objective)**
	- 장애 발생 시 **"시스템을 최대 언제까지 복구해야 하는지"** (서비스 다운타임 허용 범위)
	- RTO가 2시간이라면, 재해 발생 후 서버를 복구하여 2시간 이내에 비즈니스를 정상화

- **재해 복구에 필요한 AWS 서비스 및 기능**
	- **스토리지 복제** - S3의 교차 리전 복제, EBS의 볼륨 스냅샷 등
	- **복구용 AMI 구성** - AMI를 이용해 운영 체제 및 적절한 애플리케이션 스택으로 사전 구성
	- **장애 조치 네트워크 설계** - 로드밸런싱, 상태 확인 및 장애 조치 등
	- **데이터베이스 백업 및 복제** - 다중 AZ 배포, 자동 백업, 데이터 스냅샷 등
	- **템플릿 및 스크립트** - AWS CloudFormation 템플릿 및 스크립팅을 사용하여 리소스를 효율적으로 배포
		- CloudFormation은 인프라 구성을 텍스트 파일(JSON 또는 YAML 형식)로 적어두고 AWS에 제출하면, AWS가 알아서 자원들을 순서대로 한 방에 생성
		- 이를 **IaC (Infrastructure as Code, 코드형 인프라)**라고 부름

### AWS Backup

- **AWS Backup**은 AWS 서비스 전체에 걸쳐 데이터 백업을 중앙 집중화하고 자동화하는 데 도움이 되는 완전관리형 백업 서비스
- AWS Backup은 AWS Organizations와 연동되어, 중앙에서 데이터 보호 정책을 배포하여 백업 활동을 구성, 관리 및 제어할 수 있음

![[../80_Assets/Pasted image 20260723231834.png]]

- **AWS Backup 사용법**
	1. **Backup 계획 생성**
		- **일정** - 백업 빈도와 백업을 수행할 기간을 설정
		- **수명 주기** - 백업을 콜드 스토리지로 이동할 시기와 백업 만료 시기를 결정
		- **저장소** - 백업 계획에서 사용할 백업 저장소를 지정
		- **백업용 태그** - 이 계획을 통해 생성되는 백업에 할당할 태그를 지정
	
	2. **리소스 할당**
		- 백업 계획을 사용하여 백업할 리소스를 할당
		- 또한 AWS Backup이 이러한 리소스에 대해 필요한 액세스 권한을 얻기 위해 사용할 IAM 역할 정의
		
	3. **백업 관리 및 모니터링**
		- AWS Backup은 **Amazon CloudWatch**, **Amazon EventBridge**, **AWS CloudTrail**, **Amazon Simple Notification Service** 등의 기타 AWS 서비스와 연동되어 해당 워크로드를 모니터링

### 복구 전략

- 대부분의 기존 환경에서는 데이터를 테이프에 백업하여 정기적으로 오프사이트로 전송
- 이 방법을 사용할 경우, 가동 중단 발생 시 시스템을 복원하는 데 많은 시간이 걸릴 수 있음
![[../80_Assets/Pasted image 20260724171539.png]]

- **백업 및 복원**
  ![[../80_Assets/Pasted image 20260723233830.png]]
	- 백업은 해당 소스와 동일한 리전에서 생성되어 다른 리전에 복사
	- 백업 및 복구 전략은 RTO에서 가장 효율적이지 않은 것으로 간주

- **파일럿 라이트 (Pilot Light)**
  ![[../80_Assets/Pasted image 20260723233949.png]]
	- 파일럿 라이트 전략을 사용하면 데이터는 실시간이지만 서비스는 유휴 상태
	- 그림에서 데이터베이스는 복구 리전의 로컬 읽기 전용 클러스터에 데이터를 복제
	- 파일럿 라이트 전략에서 Elastic Load Balancing 및 Amazon EC2 Auto Scaling과 같은 기본 인프라 요소가 배치되어 있지만 컴퓨팅과 같은 기능 요소는 동작하지 않음
	  (프로덕션 장애 발생 시 가동)

- **웜 스탠바이 (Warm Standby)**
  ![[../80_Assets/Pasted image 20260723234248.png]]
	- 파일럿 라이트와 달리 동작은 하나 최소한만 동작함
	- 완전 동작 저용량 스탠바이에서는 요청을 처리할 수 있는 최소 배포가 유지되나, 프로덕션 수준의 트래픽은 처리할 수 없음
	- 장애 조치 전에 프로덕션 환경의 요구를 충족하도록 인프라를 스케일 업해야 함

- **멀티 사이트 액티브/액티브**
  ![[../80_Assets/Pasted image 20260723234417.png]]
	- 다중 사이트 액티브/액티브를 사용하면 2개 이상의 리전이 적극적으로 요청을 수락
	- 장애 조치는 요청을 처리할 수 없는 리전에서 요청을 다시 라우팅하는 것으로 구성
	- 데이터베이스 쓰기 요청의 경우 로컬 리전에 쓰기를 포함하거나 특정 리전에 쓰기를 다시 라우팅하는 등 여러 패턴을 사용할 수 있음
