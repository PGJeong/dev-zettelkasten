# 초기 아키텍처

![[../80_Assets/Pasted image 20260719153511.png]]

- 기능을 검증하고 사용자 피드백을 수집할 수 있는 간단하며 비용 효율성이 우수한 애플리케이션부터 구축하고자 함
- 이후 애플리케이션 완성도가 높아지고 수요가 증가하면, 높은 가용성과 성능을 유지하고 대규모로 확장할 수 있도록 **로드 밸런싱과 오토 스케일링, 향상된 모니터링** 방식 등의 다양한 기능을 아키텍처에 추가
- 단일 AWS 가용 영역에 개념 증명(PoC) 배포용으로 구축한 기본적인 3티어 웹 애플리케이션 아키텍처
	- 사용자는 인터넷 게이트웨이를 통해 퍼블릭 서브넷의 시스템에 액세스
	- 해당 퍼블릭 서브넷에는 Amazon EC2 웹 서버와 NAT 게이트웨이가 있음
	- 트래픽은 EC2 앱 서버가 있는 프라이빗 앱 서브넷으로 전송
	- 이 EC2 앱 서버는 데이터베이스 서브넷의 Amazon RDS 인스턴스에 연결

# 모니터링

![[../80_Assets/Pasted image 20260721192810.png]]

- 리소스가 운영되고 수행되는 상황을 추적하려면 모니터링이 필요
	- 리소스 사용률 및 애플리케이션 성능에 대한 정보를 수집
	- 인프라가 충분한지 여부를 측정
	- 인프라를 스케일 아웃 또는 스케일 인해야 하는 시기를 확인
	- 보안 측면에서 사용자가 환경의 일부에 액세스 여부 및 시기를 파악하고 권한을 확인

### Amazon CloudWatch

![[../80_Assets/Pasted image 20260721193611.png]]

- **Amazon CloudWatch**는 시스템 이벤트 스트림을 거의 실시간으로 제공하는 AWS 서비스
- CloudWatch 한곳에서 모든 AWS 리소스, 애플리케이션, 서비스에 걸쳐 이 데이터를 수집 및 액세스하고 상관관계를 파악할 수 있음
- CloudWatch는 온프레미스 서버에서도 데이터를 수집할 수 있음
- 성능 및 리소스 사용을 최적화할 수 있도록 CloudWatch에서는 자동 대시보드를 제공하며, 1초 세부 단위로 데이터를 제공하고 최대 15개월 동안 지표를 저장 및 보존

- **CloudWatch 지표**는 시스템 성능에 대한 데이터
- 대다수 서비스에서는 기본적으로 리소스 관련 지표를 제공 (EC2, EBS, RDS 등)
- EC2 인스턴스 같은 일부 리소스에 대한 세부 모니터링을 활성화하거나 자체 애플리케이션 지표를 게시할 수 있음
- CloudWatch는 검색, 그래프 처리 및 경보를 위해 계정에 모든 지표(AWS 리소스 지표 및 사용자가 제공한 애플리케이션 지표 모두)를 로드할 수 있음

### AWS CloudTrail

- **AWS CloudTrail**은 사용자 활동 및 API 사용량을 추적하여 누가 무엇을 언제 했는가에 대한 인사이트를 제공
- CloudTrail에서는 콘솔, AWS SDK, AWS CLI, 상위 수준 AWS 서비스를 통해 이루어진 호출을 포함하여 계정에 대한 AWS API 직접 호출 내역을 확인할 수 있음
- CloudTrail은 AWS API 직접 호출을 기록하고 로그 파일을 제공
- 이 정보에는 API 직접 호출자의 소스 IP 주소 및 자격 증명, API 직접 호출 시간, 요청 파라미터 및 AWS 서비스가 반환한 응답 요소가 포함
- 이를 활용하여 보안 분석, 리소스 변경 사항 추적 및 규정 준수 감사를 수행할 수 있음
	- 인스턴스가 종료된 이유가 무엇이며, 누가 종료했는가? (조직적 추적 기능 및 책임)
	- 누가 보안 그룹 구성을 변경했는가? (책임 및 보안 감사)
	- 권한 부족으로 인해 거부된 활동은 무엇인가? (네트워크에 대한 잠재적 내외부 공격)

- CloudTrail은 리전 단위로 활성화되며, 여러 리전을 사용하는 경우 리전별로 로그 파일이 전송될 장소를 선택할 수 있음. CloudTrail은 로그를 지정된 S3 버킷에 저장

| 구분           | AWS CloudTrail                    | Amazon CloudWatch                  |
| ------------ | --------------------------------- | ---------------------------------- |
| **주요 목적**    | 보안 감사, 거버넌스, 규정 준수 및 계정 활동 추적     | 리소스/애플리케이션 모니터링, 알람 설정 및 성능 최적화    |
| **핵심 질문**    | "이 EC2 인스턴스를 종료한 사람은 누구인가?"       | "이 EC2 인스턴스의 CPU 사용량이 90%를 초과했는가?" |
| **수집 데이터**   | 계정 활동, API 호출 기록, 로그인 시도 등        | 로그 파일, 메트릭(지표), 이벤트 스트림 등          |
| **주요 활용 사례** | 보안 침해 사고 조사, 규정 준수(Compliance) 보고 | 시스템 장애 감지, 자동화된 리소스 대응 조치          |
- 두 서비스는 상호 보완적으로 사용되며, 일반적으로 CloudTrail에서 수집한 보안 감사 로그를 CloudWatch Logs로 전송하여 비정상적인 API 활동이 발생할 때 실시간으로 경고를 받도록 구성하는 경우가 많음

### VPC Flow Logs

- VPC 흐름 로그는 VPC 네트워크 인터페이스에서 송수신되는 IP 트래픽 정보를 캡처
	- 이러한 로그는 VPC, 서브넷 또는 네트워크 인터페이스 기준으로 트래픽을 기록하도록 구성할 수 있음
	- EC2 및 VPC 콘솔에서 특정 리소스의 흐름 로그 탭을 선택하여 흐름 로그에 대한 정보를 볼 수 있음
	- 흐름 로그는 기본적으로 해제되어 있음. 흐름 로그 데이터를 수집하려면 옵트인해야 함

- 각 레코드는 특정 캡처 기간에 대한 네트워크 IP 흐름을 캡처하며 5가지 값(**5-tuple**)을 포함
	- 레코드에는 소스, 대상, 프로토콜 같은 IP 흐름의 구성 요소가 포함
	- 특정 유형의 트래픽이 탐지되면 활성화되는 경보를 생성하고 경향 및 패턴을 식별하는 데 도움이 되는 지표를 생성할 수 있음

### CloudWatch 경보

- CloudWatch 경보는 **지표 경보**라고 하는 단일 클라우드 지표 또는 **복합 경보**라고 하는 여러 클라우드 지표의 구성을 기반으로 생성할 수 있음
- 경보는 여러 기간 동안의 임곗값과 비교하여 지표 또는 표현식의 값을 기준으로 1개 이상의 작업을 수행
- 경보에는 3가지 상태가 있음
	- **OK** - 지표가 정의된 임곗값 내에 있음
	- **ALARM** - 모니터링되는 지표가 지정된 임곗값과 같거나, 그보다 크거나 작다는 의미
	  (e.g. 지정된 EC2 인스턴스의 CPU 사용률이 너무 높을 때를 알려 주는 경보)
	- **INSUFFICIENT_DATA** - 지정된 지표에 대한 데이터가 없을 경우

![[../80_Assets/Pasted image 20260721222837.png]]
- 위 예제에서는 경보 임곗값이 25%로 설정되어 있고 최소 위반은 2개 기간
- 즉, 임곗값이 2개 기간 연속으로 위반된 경우에만 경보 상태가 변경

### Amazon EventBridge

- **Amazon EventBridge**는 애플리케이션, 통합 SaaS 애플리케이션 및 AWS 서비스에서 생성된 이벤트를 사용하여 대규모 이벤트 기반 애플리케이션을 쉽게 구축할 수 있도록 하는 서버리스 이벤트 버스
- CloudWatch에서 캡처된 이벤트를 관리할 때는 기본적으로 EventBridge를 사용

# 로드 밸런싱

- 웹 애플리케이션은 트래픽이 가장 많은 기간에도 그리고 트래픽이 적은 기간에도 효율적으로 실행되어야 함

### Elastic Load Balancing (ELB)

![[../80_Assets/Pasted image 20260721234014.png]]

- ELB 로드 밸런서는 네이티브 방식으로 사용자를 EC2 인스턴스, 컨테이너 배포 및 AWS Lambda 함수에 연결하는 AWS에서 제공하는 유일한 로드 밸런서
- 로드 벨런서는 리전 범위 수준 서비스

- **고가용성** - ELB는 단일 가용 영역 또는 여러 가용 영역에서 다수의 대상에 걸쳐 트래픽을 자동으로 분산
- **계층 4 또는 계층 7 HTTP 및 HTTPS 로드 밸런싱** - 계층 7 전용 기능에 대해 HTTP 또는 HTTPS 애플리케이션을 로드 밸런싱 하거나, TCP만 사용하는 애플리케이션에 대해 엄격한 계층 4 로드 밸런싱을 사용할 수 있음
- **보안 기능** - VPC를 사용하면 로드 밸런서와 연결된 보안 그룹을 생성 및 관리하여 추가 네트워킹 및 보안 옵션을 제공할 수 있음. 또한 내부(인터넷에 연결되지 않음) 로드 밸런서를 생성할 수도 있음
- **상태 확인** - ELB 로드 밸런서는 비정상 대상을 감지하고, 해당 대상으로 트래픽 전송을 중단한 다음, 나머지 정상 대상으로 로드를 분산
- **모니터링 작업** - ELB는 애플리케이션 성능을 실시간으로 모니터링할 수 있도록 CloudWatch 지표와 통합되며 요청 추적 기능을 제공
