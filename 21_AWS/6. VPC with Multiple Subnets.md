# VPC (Virtual Private Cloud)

- **Amazon VPC**는 단일 AWS 리전과 연결된 가상 네트워크 (VPC는 둘 이상의 리전에 상주할 수 없음)
- **서브넷**은 1개의 가용 영역에만 상주할 수 있음 (VPC는 한 리전 내에서 둘 이상의 가용 영역에 걸쳐 있을 수 있음)
- **인터넷 게이트웨이**를 통해 VPC 리소스가 인터넷에 도달할 수 있음
- **라우팅 테이블**은 Amazon VPC 내로 들어오고 나오며 Amazon VPC 내에서 이동하는 트래픽의 라우팅을 제어
- AWS는 Amazon VPC와 온프레미스 간의 연결을 용이하게 하는 하이브리드 클라우드 구성을 지원
- AWS 계정에는 **기본 Amazon VPC**와 **사용자 지정 Amazon VPC**, 2가지 유형의 VPC가 있음

### 기본 Amazon VPC

- AWS 계정을 생성하면 지원되는 각 AWS 리전에서 기본 Amazon VPC가 생성, 고객은 AWS Cloud 내에서 환경을 즉시 구축할 수 있음
- 기본 Amazon VPC를 사용하면 리소스 배포를 즉시 시작할 수 있으며 네트워크 내부 구조에 대해 생각할 필요가 없음
- 각 기본 Amazon VPC는 지원되는 리전 내의 각 가용 영역에 퍼블릭 서브넷을 생성
- AWS는 모든 트래픽을 허용하는 구성을 설정하므로 기본적으로 개인 정보 보호 및 격리 장치가 없음

- 복원성을 높이기 위해 기본 Amazon VPC는 가용 영역 전체에서 서브넷으로 자동 분할
	- 기본 Amazon VPC는 리전당 하나만 허용
	- 각 기본 Amazon VPC는 해당 리전의 각 가용 영역에 하나의 서브넷을 갖도록 구성
	- 각 서브넷은 Amazon VPC CIDR의 IP 주소 범위의 일부를 사용
	- 각 서브넷의 IP 주소 범위는 다른 서브넷의 IP 주소 범위에 대해 고유해야 하며 겹칠 수 없음

![[../80_Assets/Pasted image 20260717152135.png]]

- 기본 Amazon VPC는 처음에 /16 IPv4 CIDR 블록, 리전의 가용 영역에 대한 /20 기본 서브넷, 인터넷 게이트웨이, 인터넷 게이트웨이에 대한 기본 라우팅, 기본 보안 그룹, 기본 네트워크 ACL 및 기본 DHCP(Dynamic Host Configuration Protocol) 옵션 집합으로 구성

### 사용자 지정 Amazon VPC

![[../80_Assets/Pasted image 20260720160552.png]]

- 기본 Amazon VPC와 달리 사용자 지정 Amazon VPC의 각 구성 요소는 생성할 때 명시적으로 정의해야 함. 명시적 구성 없이는 아무것도 허용되지 않음
- IPv4 및 IPv6 지원, VPC에 대한 CIDR 블록 등의 일부 결정은 나중에 수정할 수 없음
- 서브넷, 라우팅, VPC 엔드포인트와 같은 VPC의 다른 기능은 필요에 따라 수정할 수 있음

- 기본 Amazon VPC와 유사한 사용자 지정 Amazon VPC는 논리적으로 격리된 가상 네트워크를 제공하며, 이 네트워크는 Amazon VPC가 생성되는 리전에서 지원하는 리소스와 서비스의 배포를 지원
	- 각 서브넷을 정의하여 지원되는 IP 주소 범위 선택
	- 네트워크 내부 트래픽 흐름 관리
	- 라우팅 테이블 및 네트워크 게이트웨이를 통해 트래픽이 VPC로 들어오고 나가는 방식 관리

- 사용자 지정 Amazon VPC의 경우 기본 또는 전용 테넌시를 선택할 수 있음
	- 기본 테넌시는 다른 사용자와 공유되는 공유 하드웨어에 프로비저닝
	- 전용 테넌시는 사용자의 전용 하드웨어 (전용 테넌시는 더 많은 비용)

### 기타 사항

- 기본적으로 DNS는 Amazon VPC에 의해 처리. 그러나 Amazon Route 53을 사용하여 VPC 내에서 자체 DNS를 생성할 수도 있음
- 기본적으로 Amazon VPC의 모든 서브넷은 서로 액세스할 수 있음. 네트워크 ACL을 사용하여 서브넷으로 들어오고 나가는 트래픽을 제한할 수 있음

# 보안 및 고가용성 구성

### IAM (Identity and Access Management)

- IAM을 사용하면 VPC를 구성하고 관리할 수 있는 사용자를 제어할 수 있음
- 그룹, 사용자, 역할을 기반으로 권한 정책을 만들고 할당할 수 있음
- 권한은 지정된 리소스에 대해 각 엔터티가 수행할 수 있는 작업과 수행할 수 없는 작업을 지정할 수 있음

### 네트워크 ACL

- 네트워크 액세스 제어 목록(네트워크 ACL)은 서브넷에 들어오거나 서브넷에서 나가려고 하는 트래픽을 필터링할 수 있는 방화벽과 같은 보안 필터
- 네트워크 ACL은 서브넷 수준에서 연결(서브넷 경계를 넘는 트래픽만 관리)
- **스테이트리스(무상태)** - 들어오는 연결(인바운드)을 허용했더라도, 나가는 트래픽(아웃바운드)에 대한 규칙이 명시되어 있지 않으면 통신이 차단됨. 양방향 규칙을 각각 설정해야 함

### 보안 그룹

- 보안 그룹은 서브넷이 아닌 AWS 리소스에 연결
- 보안 그룹은 다른 보안 그룹을 참조할 수 있으며 자신을 참조할 수도 있음
- 네트워크 ACL은 허용되거나 허용되지 않는 트래픽을 명시적으로 지정하지만,
  보안 그룹 규칙은 허용되는 트래픽만 지정하며 다른 모든 트래픽을 차단
- 보안 그룹과 네트워크 ACL의 차이점은 보안 그룹이 AWS 리소스를 인식한다는 것,
  따라서 EC2 인스턴스의 경우 인스턴스 ID를 해당 인스턴스의 보안 그룹 규칙에 추가하여 인스턴스의 트래픽을 허용할 수 있음
- **스테이트풀(상태유지)** - 외부에서 들어오는 연결(인바운드)을 허용하면, 그에 대한 응답이 나가는 트래픽(아웃바운드)은 허용 여부를 따로 묻지 않고 자동으로 통과

### Elastic Load Balancer (ELB)

- 로드 밸런서에 트래픽이 들어오면 로드 밸런서는 해당 트래픽을 서버 또는 서비스에 배포
- 로드 밸런서는 AWS Auto Scaling 그룹과 페어링하여 애플리케이션의 고가용성, 내결함성, 확장성을 향상하는 데 적합
- 단일 가용 영역 또는 여러 가용 영역에서 트래픽의 다양한 로드를 처리할 수 있음

- AWS는 4가지 유형의 로드 밸런서를 제공
	- **Classic Load Balancer** - AWS의 레거시 로드 밸런서. Classic Load Balancer가 필요한 레거시 서비스 또는 애플리케이션이 없다면 사용하지 않는 것이 좋음
	- **Application Load Balancer (ALB)** - OSI 모델의 계층 7 로드 밸런서. 애플리케이션 계층, 즉 HTTP 및 HTTPs를 이해할 수 있으며 경로, 헤더, 호스트와 같은 해당 프로토콜의 항목을 기반으로 작업을 수행할 수 있음
	- **네트워크 로드 밸런서** - 네트워크 로드 밸런서는 상위 계층 프로토콜에 대해 걱정할 필요가 없고 훨씬 빠름. 고정 IP 주소를 할당할 수 있으며 보안 및 방화벽 제품과 쉽게 통합할 수 있음
	- **게이트웨이 로드 밸런서** - 방화벽, 침입 탐지 및 방지 시스템, 심층 패킷 검사 시스템과 같은 가상 어플라이언스를 배포, 크기 조정, 관리할 수 있음

![[../80_Assets/Pasted image 20260718025357.png]]
-  **VPC 기반 애플리케이션의 가용성 향상 예시**
	- 더 많은 처리량 또는 백업 옵션으로 두 번째 서브넷을 사용
	- 내결함성을 위해 두 번째 서브넷을 별도의 가용 영역에 배포하여 가용성을 극대화 (다중 AZ)
	- 하나의 라우팅 테이블을 여러 서브넷과 연결할 수 있지만, 하나의 서브넷은 하나의 라우팅 테이블만 가질 수 있음
	- Elastic Load Balancing을 사용하여 리소스 간 트래픽을 관리
	- Elastic Load Balancing을 사용하여 비정상 리소스를 감지하고 다른 서브넷의 정상 리소스로 자동으로 장애 조치

# 멀티 티어 아키텍처

- **단일 티어** - 모든 것을 하나의 서브넷에 넣음. 이렇게 하면 어떻게든 네트워크에 액세스할 수 있는 공격자에게 모든 리소스가 노출
	- 프라이빗 데이터를 사용하지 않거나, 개발자만 사용할 수 있는 경우 등에 사용
![[../80_Assets/Pasted image 20260718033946.png]]

- **멀티 티어** - 공격자와 민감한 리소스 사이에 추가 방어 계층이 존재. 실제로 로드 밸런서 또는 네트워크 주소 변환(NAT) 옵션과 같은 관리형 AWS 엔드포인트를 사용하는 경우 퍼블릭 서브넷에서 애플리케이션의 일부를 전혀 노출할 필요가 없음
	- 보안 그룹 및 네트워크 ACL을 사용하여 경로 지정되는 트래픽을 최대한 제한
![[../80_Assets/Pasted image 20260718034000.png]]

### 멀티 티어(3티어) 아키텍처 예제

![[../80_Assets/Pasted image 20260718034829.png]]
1. **계층 1: 퍼블릭 서브넷**
	- AWS는 퍼블릭 서브넷의 IP 주소 수를 프라이빗 서브넷보다 더 적게 예약하도록 권장
	- 인터넷에 대한 노출을 제한하기 위해, 들어오는 트래픽은 Application Load Balancer가, 나가는 트래픽은 NAT 솔루션이 처리
	- 로드 밸런서 및 NAT 게이트웨이는 Managed Services이며 기본적으로 고가용성이므로 이러한 서비스에서 병목 현상이 발생할 염려가 없음

2. **계층 2: 프라이빗 서브넷의 애플리케이션**
	- Amazon EC2 인스턴스를 실행하기 위한 프라이빗 서브넷 계층
	- 각 서브넷에는 각 애플리케이션의 크기 조정 요구 사항을 수용하기 위해 1,024개의 IP 주소가 예약
	- 두 퍼블릭 서브넷 모두에 연결된 Application Load Balancer는 프라이빗 서브넷의 애플리케이션 리소스 간에 트래픽을 분산

3. **계층 3: 프라이빗 서브넷의 데이터**
	- 데이터 리소스를 첫 번째 프라이빗 서브넷 뒤에 있는 두 번째 프라이빗 서브넷에 배치
	- 데이터 계층은 Amazon RDS 배포이거나 EC2에서 실행되는 데이터베이스일 수 있음
	- 다중 AZ 구성을 사용

- **트래픽 이동 경로:**
  인터넷 게이트웨이 > Application Load Balancer > 애플리케이션 > 데이터 > 애플리케이션 > NAT 솔루션 > 인터넷 게이트웨이
