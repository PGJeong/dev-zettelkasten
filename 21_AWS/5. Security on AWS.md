- **예방** - 사용자 권한 및 자격 증명, 인프라 보호 및 데이터 보호 수단을 정의
- **탐지** - 로깅 및 모니터링 서비스를 통해 조직의 보안 태세를 파악
- **대응** - 자동화된 인시던트 대응 및 복구
- **문제 해결** - 이벤트 기반 자동화 기능을 활용하여 빠른 속도로 AWS 환경 보호 및 문제 해결

# 보안 설계 원칙

- **최소 권한 원칙** - 실제로 필요로 하는 사용자에게만 데이터 및 기타 리소스에 대한 액세스 권한 부여
- **추적 기능 구현** - 환경에서 수행되는 작업과 변경 사항을 실시간으로 모니터링, 알림 및 감사

- **전송 중 데이터 & 저장 데이터 보호** - AWS는 저장된 데이터와 전송 중인 데이터를 보호할 수 있도록 서비스와 기능을 제공
- **보안 이벤트 대비** - 보안 인시던트가 발생하기 전에 미리 도구와 액세스 권한을 갖춤, 게임 데이를 통해 정기적으로 인시던트 대응 방법을 연습
- **공격 대상 영역 최소화** - 규모를 조정하여 공격을 흡수하거나 영향을 최소화하거나 디바이스가 보호되지 않을 가능성을 최소화하여 대비

# 공동 책임 모델

- 보안 및 규정 준수를 관리하는 것은 AWS와 고객의 공동 책임

![[../80_Assets/Pasted image 20260705231346.png]]

### AWS의 책임

- AWS는 클라우드 자체의 보안을 담당
- AWS 리전, 가용 영역, 데이터 센터 보호, 건물의 물리적 보안
- AWS 서비스를 실행하는 하드웨어, 소프트웨어, 네트워킹 구성 요소 관리 (물리적 서버, 호스트 운영 체제, 가상화 계층, AWS 네트워킹 구성 요소 등)
- AWS의 책임 수준은 서비스에 따라 달라짐

| 카테고리        | 카테고리 내 AWS 서비스의 예                                            | **AWS의 책임**                                            |
| ----------- | ------------------------------------------------------------ | ------------------------------------------------------ |
| **인프라 서비스** | 컴퓨팅 서비스. 예: Amazon Elastic Compute Cloud(Amazon EC2)         | AWS는 기본 인프라 및 기초 서비스를 관리합니다.                           |
| **추상화 서비스** | 고객의 관리가 거의 필요하지 않은 서비스. 예: Amazon Simple Storage Service(S3) | AWS는 서버 측 암호화 및 데이터 보호 외에도 인프라 계층, 운영 체제 및 플랫폼을 운영합니다. |

### 고객의 책임

- 고객은 클라우드 내부의 보안을 책임짐
- 고객은 데이터 보안을 보장할 뿐만 아니라 서비스 및 애플리케이션을 올바르게 구성할 책임
- 고객의 책임 수준은 AWS 서비스에 따라 달라짐
- 일부 서비스의 경우 필요한 모든 보안 구성 및 관리 작업을 고객이 수행해야

| **카테고리**    | **카테고리 내 AWS 서비스의 예**                                        | **고객 책임**                                                          |
| ----------- | ------------------------------------------------------------ | ------------------------------------------------------------------ |
| **인프라 서비스** | 컴퓨팅 서비스. 예: Amazon Elastic Compute Cloud(Amazon EC2)         | 고객은 고객 데이터를 암호화, 보호, 관리하는 것 외에도 운영 체제 및 애플리케이션 플랫폼을 제어합니다.         |
| **추상화 서비스** | 고객의 관리가 거의 필요하지 않은 서비스. 예: Amazon Simple Storage Service(S3) | 고객은 고객 데이터를 관리하고, 데이터를 암호화하고, 네트워크 방화벽 및 백업을 통해 데이터를 보호할 책임이 있습니다. |

- 데이터 주권 규정에 따라 AWS 리소스의 리전 선택
- 암호화 기법, 일정에 따른 백업 등 데이터 보호 메커니즘 구현
- 데이터와 AWS 리소스에 액세스할 수 있는 사람을 제한하기 위해 액세스 제어 사용

# AWS 루트 사용자

- AWS 계정을 처음 생성할 때는 결제 및 개인 정보를 비롯하여 계정의 모든 AWS 서비스 및 리소스에 대해 완전한 액세스 권한을 갖는 단일 로그인 자격 증명으로 시작
- 이 자격 증명은 AWS 루트 사용자라고 하며, 계정을 생성할 때 사용한 이메일 주소 및 암호로 로그인하여 액세스

- 루트 사용자와 관련된 자격 증명을 안전하게 보관하고 일상 작업에는 루트 사용자를 사용하지 않아야 함
- 관리 작업이나 일상 작업용으로 Identity and Access Management(IAM) 사용자를 생성

### 자격 증명

- AWS 루트 사용자에게는 2가지 자격 증명 집합이 연결됨
	- **이메일 주소와 암호** - 이를 통해 AWS Management Console에 액세스할 수 있음
	- **액세스 키** - 이 키를 사용해서 AWS CLI 또는 AWS API에서 프로그래밍 방식으로 요청을 수행할 수 있음 (액세스 키는 필요할 때만 생성할 것)

- 엑세스 키는 다음 두 부분으로 구성
	- **액세스 키 ID:** 예를 들어, `A2lAl5EXAMPLE`
	- **비밀 액세스 키:** 예를 들어, `wJalrFE/KbEKxE`
- CLI 또는 API를 통해 요청을 인증하려면 액세스 키 ID와 비밀 액세스 키를 함께 사용

- MFA에서는 2개 이상의 인증 방법을 사용하여 자격 증명을 확인 (MFA 디바이스에서 제공하는 임시 숫자 코드)

# AWS IAM (Identity and Access Management)

- IAM은 AWS 리소스에 대한 액세스를 관리하는 데 도움이 되는 웹 서비스
- IAM을 사용하면 권한이 부여된 사용자 및 애플리케이션만 AWS 서비스 및 리소스와 상호 작용하도록 허용
- IAM은 인증(로그인)했으며 리소스 사용 권한이 부여(권한 소유)된 사용자를 확인하는 방식으로 각 리소스에 액세스 가능한 사용자를 제어
- IAM은 특정 AWS 리전만이 아닌 전 세계적으로 적용

### 엑세스 권한 받기

1. 사용자 또는 애플리케이션 인증
2. 액세스 요청에 대해 권한 부여
3. AWS 리소스에서 작업을 수행하는 권한 부여

### IAM 구성 요소

- **사용자** - AWS와 상호 작용하는 사람 또는 애플리케이션
- **그룹** - 필요한 액세스 권한이 비슷한 IAM 사용자 모음
- **역할** - 사용자 또는 서비스가 수임할 수 있는 권한 세트
- **정책** - 지정된 리소스에 대한 작업을 허용하거나 거부하는 방식으로 권한을 정의한 JSON 문서

### IAM 사용자

- AWS 계정 내에서 정의되는 IAM 사용자에게는 AWS 서비스 및 리소스 액세스와 관리를 위한 특정 권한이 부여
- 필요에 따라 이러한 사용자를 생성, 수정 및 삭제할 수 있음

- 일반적으로 각 IAM 사용자에게는 본인의 보안 자격 증명 세트가 지정
- 이러한 자격 증명에는 콘솔 액세스용 **사용자 이름과 암호**, 프로그래밍 방식 AWS 서비스 액세스용 **액세스 키** 등이 있음

- IAM 사용자를 생성할 때 다음과 같은 여러 설정을 구성할 수 있음

| 설정             | 정의                                                        |
| -------------- | --------------------------------------------------------- |
| **사용자 이름**     | AWS 계정 내 사용자의 고유 식별자입니다.                                  |
| **액세스 유형**     | 사용자가 AWS와 상호 작용하는 방식(프로그래밍 방식 액세스, 콘솔 액세스 또는 둘 다)을 결정합니다. |
| **권한**         | 사용자가 각 리소스에서 수행할 수 있는 작업을 정의합니다.                          |
| **암호 정책**      | 암호 복잡성 및 만료 관련 요구 사항을 설정합니다.                              |
| **다중 인증(MFA)** | 사용자 로그인용 보안 계층을 더 추가합니다.                                  |
| **태그**         | 구성 및 관리용으로 사용자에게 연결할 수 있는 메타데이터입니다.                       |

### 고유 사용자 유형

- **루트 사용자** - 모든 서비스에 대한 전체 권한이 있는 기본 계정 소유자입니다. 광범위한 권한이 부여되는 사용자 유형이므로 최소한으로만 사용해야 합니다.
- **페더레이션 사용자** - 외부 자격 증명 공급자(IdP)를 통해 AWS 외부에서 인증을 진행하여 AWS 리소스에 액세스하는 사용자입니다.

### IAM 정책

- JSON 형식으로 작성되는 IAM 정책은 사용자, 그룹 또는 역할이 특정 AWS 리소스에서 수행할 수 있는 작업을 정의 및 제어
- IAM 정책은 IAM 사용자, IAM 그룹 또는 IAM 역할에 연결되어야 함
- AWS 리소스에 대한 액세스 권한을 요청하면 AWS에서는 평가 로직을 사용하여 해당 요청의 허용 여부를 결정
- AWS는 필요에 따라 할당할 수 있는 사전 구성된 정책인 AWS 관리형 정책을 제공

- 정책에는 필수 요소와 선택적 요소가 있음

|요소|설명|필수|
|---|---|---|
|**효과**|Allow 또는 Deny를 사용하여 정책이 액세스를 허용하는지 아니면 거부하는지를 나타냅니다.|예|
|**보안 주체**|액세스를 허용하거나 거부할 계정, 사용자, 역할 또는 페더레이션 사용자를 나타냅니다(리소스 정책에만 해당됨).|아니요|
|**작업**|정책이 허용하거나 거부하는 작업 목록을 포함합니다.|예|
|**리소스**|작업이 적용되는 리소스 목록을 지정합니다.|예|
|**조건**|정책이 권한을 부여하는 조건을 지정합니다.|아니요|

- 계정 내 EC2 인스턴스 보기 권한을 제공하는 자격 증명 기반 IAM 정책 예시
```json
{
 "Version": "2012-10-17",
 "Statement": [
  {
    "Effect": "Allow",
    "Action": "ec2:DescribeInstances",
    "Resource": "*"
  }
 ]
}
```

- **"Version": "2012-10-17":** 사용 중인 정책 언어 버전을 표시하기 위해 AWS에서 사용하는 표준 날짜
- **"Statement":** 권한을 정의하는 정책의 주요 부분
- **"Effect": "Allow":** 정책이 권한을 거부하는 것이 아니라 부여함을 지정
- **"Action": "ec2:DescribeInstances":** 허용할 구체적인 작업. Action 뒤의 ec2:DescribeInstances는 EC2 인스턴스 관련 세부 정보 보기 권한을 제공한다는 의미
- **"Resource": " ":** 별표는 계정 내 모든 EC2 계정에 해당 권한이 적용됨을 의미

### 정책 유형

- 정책은 자격 증명 또는 리소스에 연결될 수 있음

| 정책 유형           | 설명                                                                                                                                                                                                                           |
| --------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **자격 증명 기반 정책** | 관리형 및 인라인 정책을 IAM 자격 증명(사용자, 사용자가 속한 그룹 및 역할)에 연결합니다.                                                                                                                                                                        |
| **리소스 기반 정책**   | 리소스에 인라인 정책을 연결합니다. 리소스 기반 정책의 가장 일반적인 예는 Amazon Simple Storage Service(S3) 버킷 정책 및 IAM 역할 신뢰 정책입니다.                                                                                                                         |
| **조직 SCP**      | AWS Organizations 사용 시 서비스 제어 정책(SCP - Service Control Policy)을 사용하여 조직 또는 조직 단위(OU)의 멤버 계정에 대한 최대 권한을 정의합니다.                                                                                                                |
| **ACL**         | 액세스 제어 목록(ACL)을 사용하여 ACL이 연결된 리소스에 액세스할 수 있는 다른 계정의 보안 주체를 제어합니다. ACL도 리소스 기반 정책과 비슷하지만 JSON으로 작성되지 않습니다. ACL의 형식은 서비스에 따라 다릅니다(Amazon S3의 경우 XML, 네트워크 ACL의 경우 규칙 테이블). ACL은 별도의 정책 파일로 저장되는 것이 아니라 보호 대상 리소스와 함께 직접 저장됩니다. |

### IAM 역할 (임시 액세스용)

- IAM 역할은 특정 엔터티가 AWS 환경 내에서 수행할 수 있는 작업을 결정하는 권한 세트
- 역할은 AWS Management Console을 통해 구성할 수도 있고, 다양한 AWS SDK 또는 CLI 도구를 사용하여 프로그래밍 방식으로 구성할 수도 있음

- 일반적으로는 영구 자격 증명을 사용할 필요 없이 AWS 리소스에 대한 **임시 액세스 권한**을 부여하려는 경우 역할을 생성
- 관리자는 IAM 역할을 정의할 때 해당 역할을 수임할 수 있는 엔터티(예: AWS 서비스, 애플리케이션 또는 사용자)를 식별하는 신뢰 정책을 지정

| 비교 항목                  | IAM 사용자 (IAM User)               | IAM 역할 (IAM Role)               |
| ---------------------- | -------------------------------- | ------------------------------- |
| **개념**                 | AWS 내의 특정 **사람 또는 단일 앱**         | 누구나 필요할 때 **임시로 탈 수 있는 모자(자격)** |
| **자체 권한 보유 여부**        | **가능** (사용자에게 정책 직접 부여 가능)       | **가능** (역할 자체에 정책 부여 가능)        |
| **인증 방식 (Identity)**   | 로그인 ID/PW, 영구 액세스 키 (Access Key) | 자격 증명 없음 (다른 주체의 인증에 의존)        |
| **인가 방식 (Permission)** | 사용자에게 연결된 정책에 의해 영구적 유지          | 역할을 맡는 동안만 임시 토큰(STS)으로 유지      |
| **권한 획득 과정**           | 로그인하면 부여된 권한을 즉시 사용              | `AssumeRole` 조치를 통해 역할을 위임받아 사용 |
| **보안성**                | 키 유출 시 영구적 위험 (주기적 로테이션 필요)      | 유출되어도 수 분~수 시간 내 만료되므로 안전       |
| **추천 사용 사례**           | 일상적인 콘솔 로그인, 로컬 PC CLI 환경        | AWS 서비스 간 통신(EC2→S3), 교차 계정 접근  |

### IAM을 사용하는 페더레이션 계정 액세스

![[../80_Assets/Pasted image 20260716132235.png]]
- AWS ID 페더레이션을 구현하면, 자격 증명 공급자(IdP)를 통해 사용자 인증을 진행할 수 있음

- 구현 순서
	1. AWS에서 각 병원 네트워크용으로 IAM IdP 생성
	2. 페더레이션 사용자용 IAM 역할 및 권한 생성
	   (의료 분야 직무(의사, 간호사, 관리자)용 IAM 역할을 생성)
	3. AWS 역할에 사용자 속성 매핑
	   (직함이나 부서 등의 자격 증명 속성에 따라 AWS 역할을 지정)
	4. 사용자 액세스 활성화
	   (각 IdP 내에서 직무와 액세스 요구 사항에 따라 특정 병원의 사용자와 그룹을 해당하는 IAM 역할에 매핑하는 어설션을 정의)

# 액세스 패턴 구현

### 동일 계정 서비스 간 통신

- 단일 AWS 계정 내에서 여러 서비스가 통신할 때 IAM에서 제공되는 2가지 주요 방식으로는,
  **리소스 정책을 통한 액세스 제어**와 **IAM 역할을 통한 액세스 제어**가 있음

- **리소스 정책을 통한 액세스 제어** - Amazon S3, Amazon Simple Notification Service(SNS), AWS Lambda 등 리소스 기반 정책을 지원하는 서비스의 경우 리소스에 권한을 직접 연결
	- **예제**
		1. S3 버킷 리소스 정책 등의 리소스 정책을 리소스에 직접 연결
		2. 해당 리소스에 액세스할 수 있는 보안 주체(서비스에서 사용하는 IAM 역할 포함)를 지정

![[../80_Assets/Pasted image 20260716223247.png]]

- **IAM 역할을 통한 액세스 제어** - 다른 AWS 서비스에서 작업을 수행하기 위해 AWS 서비스가 수임할 수 있는 IAM 역할(적절한 권한이 적용된 역할)을 생성
	- **예제**
		1. 특정 리소스 액세스 권한이 적용된 IAM 역할을 생성
		2. 해당 권한이 필요한 서비스(예: Lambda)에 역할을 연결
		3. 다른 리소스에 액세스해야 할 때 이 역할을 사용하도록 서비스를 구성

![[../80_Assets/Pasted image 20260716223705.png]]

### 크로스 계정 서비스 간 통신

- AWS 서비스가 다른 AWS 계정의 리소스에 액세스해야 하는 경우에는 크로스 계정 액세스 제어를 구현해야 함
- 이 액세스 방식에도 동일 계정 액세스 방식의 개념이 적용되지만 계정 간의 신뢰 관계가 추가로 적용됨

- **크로스 계정 액세스 작동 방식**
	- 리소스 계정(액세스하려는 리소스가 포함된 계정)은 서비스 계정(액세스 권한을 필요로 하는 서비스가 포함된 계정)에 권한을 부여해야 함
	- 크로스 계정 리소스에 액세스하도록 서비스 계정의 서비스를 구성해야 함

- **리소스 정책을 통한 크로스 계정 액세스**
	- **예제**
		1. 계정 B의 S3 버킷에 적용된 리소스 정책은 계정 A의 Lambda 함수 실행 역할에 액세스 권한을 부여
		2. 계정 A의 Lambda 실행 역할에는 최소한의 Amazon S3 권한만 있으면 되며, 크로스 계정 역할은 수임할 필요가 없음

![[../80_Assets/Pasted image 20260716224752.png]]

- **IAM 역할을 통한 크로스 계정 액세스**
	- **예제**
		1. 리소스 계정(계정 B)에서 역할을 생성
		2. 서비스 계정(계정 A)의 역할 수임을 허용하도록 신뢰 정책을 구성
		3. 필요한 리소스 액세스 권한을 부여하는 권한 정책을 해당 역할에 연결
		4. 서비스 계정에서 크로스 계정 역할 수임 권한을 서비스에 부여
		5. 리소스에 액세스할 때 해당 역할을 수임하도록 서비스를 구성

![[../80_Assets/Pasted image 20260716225238.png]]

| 크로스 계정 리소스 정책             | 크로스 계정 IAM 역할         |
| ------------------------- | --------------------- |
| 구현이 더 단순함                 | 모든 AWS 서비스에서 사용 가능    |
| 역할 수임 없이 직접 액세스 제공        | 더욱 중앙 집중식으로 권한 관리 가능  |
| 리소스 정책을 지원하는 서비스에서만 사용 가능 | 수명 제한이 있는 임시 자격 증명 제공 |
| 특정 권한 부여 시 적합             | 복잡한 권한 시나리오에 적합       |
- 보안 요구 사항에 따라 **리소스 정책**과 **IAM 역할 수임** 중 적합한 방식을 선택해야 함
