- AWS 글로벌 클라우드 인프라는 전 세계 데이터 센터를 통해 완전한 기능을 갖춘 200가지가 넘는 서비스를 제공
- AWS에서 물리적 인프라는 Region(리전) 및 Availability Zones(가용 영역)의 형태로 AWS 글로벌 인프라를 구성

# AWS Infrastructure

AWS 서비스를 실행할 수 있는 물리적 위치

- **Data centers**
	- 실제 서버를 호스팅하는 비공개 시설
	
- **Availability Zones(AZ, 가용 영역)**
	- AWS 리소스를 가동하는 최소 단위
	- 특정 지리적 위치 내의 격리된 여러 영역
	- 여러 가용 영역에 인스턴스를 배포하고 인스턴스 1개에 장애가 발생하는 경우 다른 가용 영역의 인스턴스가 요청을 처리할 수 있도록 애플리케이션을 설계할 수 있음
	- 데이터 센터의 클러스터
	
- **Regions(리전)**
	- 1개의 지리적 영역 내에서 격리되고 물리적으로 분리된 여러 개의 가용 영역으로 구성, 이를 통해 가장 강력한 내결함성 및 안정성을 달성
	- 각 리전은 완전히 독립적
	- AZ의 클러스터
	- 해당 리전의 위치와 최종 사용자 간의 지연 시간, 해당 리전의 리소스 비용, 거버넌스, 해당 리전의 특정 서비스 사용가능 여부를 고려하여 리전 선택

	```
	us-east-1 = AWS 리전의 이름
	us-east-1a = 뒤에 나오는 문자는 가용영역(AZ)
	서비스를 AZ에 배포할 때가 있고, 리전에 배포할 때도 있음(서비스 범위 수준에 따라)
	```
	
- **AWS Local Zones(로컬 영역)**
	- 사용할 수 있는 AWS 서비스 수가 제한된 특정 위치
	- 컴퓨팅, 스토리지 서비스와 같은 엄선된 AWS 서비스가 더 많은 최종 사용자에게 더 가까운 위치에서 제공
	- 최종 사용자에게 10밀리초 미만의 지연 시간이 필요한 수요가 많은 애플리케이션에 사용
	- 사용자와 호스트 위치 사이의 지연 시간이 짧아야 하는 워크로드에 적합
	
- **Edge locations(엣지 로케이션)**
	- AWS 서비스 요청자에 가장 가까이 있는 지점
	- 전 세계 주요 도시에 위치
	- 요청을 수신하고 더 빠른 전송을 위해 콘텐츠 사본을 캐시

![[../80_Assets/Pasted image 20260705225912.png]]
	- 남아메리카의 Amazon S3에 저장된 동영상 파일의 예
	- 파일이 고객과 가까운 엣지 로케이션에 캐시되어 아시아의 고객에게 빠르게 제공

# 네트워킹 기본 사항

- IP 주소는 네트워크 내 리소스의 위치를 식별
- 이 주소는 2개의 부분(네트워크, 호스트)으로 구성되며, 한 부분은 네트워크를 식별하고 다른 부분은 네트워크 내의 호스트를 식별

- **IPv4 주소** - 32비트 주소로, 8비트(옥텟) 집합 4개로 그룹화
- **IPv6 주소** - 128비트 주소로, 4개의 16진수 자릿수로 구성된 8개 그룹

![[../80_Assets/Pasted image 20260706161306.png]]

- 슬래시(/) 숫자가 클수록 네트워크는 작음

### CIDR (Classless Inter-Domain Routing) 표기법

- 숫자와 점을 사용하는 표기법으로 네트워크를 식별, 슬래시 표기법으로 서브넷 마스크를 지정
- 서브넷 마스크가 /16인 경우 첫 16비트(옥텟 2개)가 네트워크 식별용으로 예약, 나머지 16비트가 호스트 식별에 사용
- AWS에서 /28(16개 IP 주소) ~ /16(65,536개 IP 주소) 사이에서 블록 크기를 지정
- Amazon VPC는 IPv4 및 IPv6 주소 지정을 지원하며, 이들 IP에 대해 상이한 CIDR 블록 크기 제한이 있음
- 기본적으로 모든 VPC와 서브넷에는 IPv4 CIDR 블록이 있어야 함

![[../80_Assets/Pasted image 20260706184807.png]]

### 서브넷 마스크

- 호스트가 로컬 서브넷에 있는지 확인하는 데 사용되는 32비트 숫자
- 호스트 비트를 모두 0으로 설정하고 네트워크 비트를 모두 1로 설정하여 생성

![[../80_Assets/Pasted image 20260706185010.png]]

### 탄력적 IP 주소 이동

 - 탄력적 IP 주소를 인스턴스 또는 네트워크 인터페이스와 연결할 수 있음
 - 탄력적 IP 주소를 사용하면 주소를 VPC의 다른 인스턴스에 신속하게 다시 매핑하여 인스턴스의 실패를 마스킹할 수 있음
	 - 이 예에서는 탄력적 IP 주소 203.0.133.15를 사용하는 EC2 인스턴스에서 장애가 발생하여, 이 탄력적 IP 주소는 새 EC2 인스턴스에 배정됨

# Amazon Virtual Private Cloud (VPC)

- Amazon VPC는 클라우드의 네트워크 환경, AWS 계정 전용 가상 네트워크
- 사용자가 정의한 가상 네트워크 안에서 AWS 리소스를 시작할 수 있음
- VPC는 AWS 리전 중 1개에 배포되며, 해당 리전 내의 모든 가용 영역에서 리소스를 호스팅할 수 있음
- IPv4와 IPv6를 모두 사용하여 리소스와 애플리케이션에 안전하고 쉽게 액세스
- VPC에 할당한 CIDR 블록은 수정할 수 없다
	- 확장을 위해서는 보조 CIDR 블록을 추가할 수 있음 (최대 5개)
	- 또는 기존 VPC를 삭제하고 새 VPC를 생성해 마이그레이션

### 서브넷

- 서브넷은 VPC 내의 IP 주소 범위로, AWS 리소스를 지정된 서브넷에서 시작할 수 있음
- 라우팅 정책을 위한 컨테이너
- 인터넷에 연결되어야 하는 리소스에는 **퍼블릭 서브넷**을 사용, 인터넷에 연결되지 않는 리소스에는 **프라이빗 서브넷**을 사용
- 서브넷은 1개의 가용 영역(AZ) 내에 상주 (다른 가용 영역으로 확장 불가)
- AWS는 각 서브넷의 CIDR 블록에서 첫 IP 주소 4개와 마지막 IP 주소를 예약 (5개 주소 예약)

![[../80_Assets/Pasted image 20260707164851.png]]

- 각 서브넷은 라우팅 테이블과 연결되어야 함
	- 생성하는 모든 서브넷은 자동으로 VPC의 기본 라우팅 테이블과 연결
- 퍼블릭 서브넷은 인터넷 게이트웨이 측 경로가 있는 라우팅 테이블에 연결되어, 퍼블릭 IP 주소를 할당하여 퍼블릭 인터넷에서 해당 서브넷 내부의 리소스에 도달할 수 있음
- **퍼블릭 IP 주소**는 인터넷에서 액세스할 수 있는 주소
	- 인터넷에 대한 인바운드 및 아웃바운드 액세스를 지원하도록 인터넷 게이트웨이에 대한 라우팅 테이블 항목이 있음
- **프라이빗 IP 주소**는 네트워크에서만 연결할 수 있는 주소
	- 인터넷 게이트웨이에 대한 라우팅 테이블 항목이 없음 (인터넷에서 직접 엑세스 불가)
	- 일반적으로 NAT 게이트웨이를 사용하여 제한된 아웃바운드 인터넷 액세스 지원

![[../80_Assets/Pasted image 20260707165200.png]]

- IPv4 전용 서브넷, IPv6 전용 서브넷, 듀얼 스택 IPv4 및 IPv6 서브넷을 설정할 수 있음
	- IPv4 통신과 IPv6 통신은 서로 독립적

### 라우팅 테이블

- **라우팅 테이블**은 네트워크 트래픽이 전달되는 위치를 결정하는 데 사용되는 규칙 세트(경로)를 포함
- VPC를 생성하면, 자동으로 기본 라우팅 테이블이 생성
- 처음에는 단일 경로만 기본 라우팅 테이블(및 VPC의 모든 라우팅 테이블)에 포함하며, 이는 VPC 내의 모든 리소스에 대한 통신을 허용하는 로컬 경로
- VPC에서 인스턴스를 시작할 때마다 로컬 경로는 그 인스턴스를 자동으로 포함
- 듀얼 스택 VPC의 경우 IPv4 트래픽과 IPv6 트래픽은 별도로 처리
![[../80_Assets/Pasted image 20260707173008.png]]

- 각 AWS 계정에는 즉시 사용 가능하도록 미리 구성된 기본 VPC가 포함, 자체적으로 VPC를 생성하고 구성할 필요가 없음

### 인터넷 게이트웨이 (IGW)

- **인터넷 게이트웨이**는 VPC 내 인스턴스와 인터넷 간의 통신을 허용
- 인터넷 게이트웨이는 퍼블릭 IP 주소와 프라이빗 IP 주소를 매핑하여 Network Address Translation(NAT)을 수행
	- 이 예에서 인터넷 게이트웨이는 요청의 소스 IP 주소를 네트워크에서 사용되는 프라이빗 IP 주소(172.31.2.15)에서 퍼블릭 IP 주소(54.56.9.10)로 변환
- 수평으로 규모가 조정되고 중복성이 있으며 가용성이 뛰어난 VPC 구성 요소

### NAT 게이트웨이

![[../80_Assets/Pasted image 20260717121419.png]]
- **NAT 게이트웨이**는 AWS 관리형 서비스로서 프라이빗 서브넷의 인스턴스는 NAT 게이트웨이를 통해 인터넷이나 다른 AWS 서비스에 연결할 수 있지만 인터넷은 해당 인스턴스에 대한 연결 불가
	- 프라이빗 서브넷에서 오는 트래픽은 프라이빗 서브넷의 라우팅 테이블에 지정된 대로 NAT 게이트웨이로 라우팅
	- 그러면 NAT 게이트웨이가 트래픽을 인터넷 게이트웨이로 전달하여 통신의 중개자 역할

- NAT 게이트웨이는 올바르게 작동하려면 공용 서브넷에 설치해야 함. 위 다이어그램에서는 NAT 게이트웨이가 공용 서브넷에 설치되어 사설 서브넷의 리소스가 인터넷에 접근할 수 있도록 함
- NAT 게이트웨이의 고가용성은 단일 가용 영역으로 제한되며, 가용 영역 전체의 고가용성을 위해서는 서로 다른 가용 영역에 최소 2개의 NAT 게이트웨이를 두는 것이 좋음

![[../80_Assets/Pasted image 20260720223109.png]]
- NAT는 메시지의 한 IP 주소를 다른 IP 주소에 매핑하여, 사용되는 IP 주소를 보호
- NAT에서는 프라이빗 IP 주소가 퍼블릭 IP 주소에 매핑되므로 NAT를 사용하면 프라이빗 IP 주소의 인터넷 연결을 허용할 수 있음

| 구분         | 인터넷 게이트웨이 (IGW)         | NAT 게이트웨이              |
| ---------- | ----------------------- | ---------------------- |
| **통신 방향**  | **양방향** (외부 인터넷과 직접 연결) | **단방향** (내부에서 외부로만 통신) |
| **위치**     | 퍼블릭 서브넷                 | 퍼블릭 서브넷                |
| **대상 리소스** | 공인 IP를 가진 리소스           | 사설 IP만 가진 리소스          |
| **보안/접근**  | 인터넷에서 내부로 직접 접근 가능      | 외부에서 내부로 접속 불가 (보안 우수) |

### 송신 전용 인터넷 게이트웨이

- IPv6 트래픽에만 사용, 본질적으로 스테이트풀
- VPC 내의 인스턴스에서 인터넷으로의 아웃바운드 IPv6 통신을 허용하는 동시에,
  인터넷에서 시작되는 인바운드 트래픽을 차단
- 보안 그룹을 연결할 수는 없음, 그러나 인스턴스에 대한 보안 그룹을 사용하여 해당 인스턴스와 주고받는 트래픽을 제어할 수는 있음

# VPC Security

- 여러 개의 독립적인 보안 계층이 저지선으로 작용하여 공격의 모멘텀과 효과를 축소
- 2가지 주요 보안 제약 조건인 **네트워크 ACL**과 **보안 그룹**

### NACL(네트워크 액세스 제어 목록)

- 네트워크 ACL은 1개 이상의 서브넷에서 보내고 받는 트래픽을 제어하기 위한 방화벽 역할을 하는 VPC에 대한 선택적 보안 계층
- 서브넷 수준에서 인바운드 트래픽과 아웃바운드 트래픽을 모두 제어
- 모든 VPC에는 수정 가능한 기본 네트워크 ACL이 자동으로 제공되며, 이 ACL은 모든 인바운드 및 아웃바운드 IPv4 트래픽을 허용
- **스테이트리스(비상태형)** - 들어오는 연결(인바운드)을 허용했더라도, 나가는 트래픽(아웃바운드)에 대한 규칙이 명시되어 있지 않으면 통신이 차단됨. 양방향 규칙을 각각 설정해야 함
- 네트워크 ACL 규칙은 순서가 중요(규칙 번호가 가장 낮은 항목부터 시작하여 차례로 평가)

### 보안 그룹

- 보안 그룹은 인스턴스에 대한 인바운드 및 아웃바운드 트래픽을 제어하는 가상 방화벽 역할
- 인스턴스 수준에서 인바운드 및 아웃바운드 트래픽을 모두 제어
	- 프로토콜, 포트, IP 주소를 기반으로 인바운드 및 아웃바운드 규칙을 정의
	- 특정 리소스에 대한 액세스를 허용하거나 거부할 수 있음
- **스테이트풀(상태저장형)** - 외부에서 들어오는 연결(인바운드)을 허용하면, 그에 대한 응답이 나가는 트래픽(아웃바운드)은 허용 여부를 따로 묻지 않고 자동으로 통과
- 여러 보안 그룹을 1개의 인스턴스에 연결하여 계층화된 보안 모델을 생성하고 심층적인 방어 전략을 구현할 수 있음

- **보안 그룹 체인** - 트래픽이 상위 티어에서 하위 티어로만 흐른 후 다시 반대로 흐를 수 있도록 인바운드 및 아웃바운드 규칙이 설정 (보안 그룹이 방화벽 역할을 수행)

# Amazon EC2 (Elastic Compute Cloud)

- Amazon EC2는 가상 머신을 생성하고 실행하는 데 사용하는 서비스
- 온프레미스 서버와 동일하지만 클라우드에서 제공
- EC2는 웹 호스팅, 애플리케이션, 데이터베이스, 인증 서비스 등의 워크로드와 서버가 지원할 수 있는 기타 모든 워크로드를 지원

### Amazon Machine Image (AMI)

- 클라우드의 가상 서버인 인스턴스를 시작하는 데 필요한 정보를 제공
- 동일한 구성의 인스턴스가 여러 개 필요할 때는 한 AMI에서 여러 인스턴스를 시작할 수 있음
- 서로 다른 구성의 인스턴스가 필요할 때는 다양한 AMI를 사용하여 인스턴스를 시작할 수 있음
- 인스턴스를 시작할 때 소스 AMI를 지정
- AWS에서 제공하는 AMI를 사용할 수도 있고 사용자 정의 AMI를 직접 생성할 수도 있음

### 인스턴스 유형

- 각 인스턴스 유형의 크기와 할당되는 가상 CPU(vCPU) 및 메모리는 서로 다름
- 인스턴스를 효율적으로 사용하려면 적절한 인스턴스 크기를 선택해야 함

![[../80_Assets/Pasted image 20260707223457.png]]
![[../80_Assets/Pasted image 20260707223528.png]]

- **AWS Compute Optimizer**는 기계 학습을 사용하여 Amazon CloudWatch에서 제공되는 사용 데이터와 리소스의 현재 구성을 분석, 구성과 사용량 분석 결과에 따라 권장 컴퓨팅이 제공

### Amazon EC2 키 페어

- 프라이빗 키와 퍼블릭 키로 구성되어 있는 키 페어는 인스턴스에 연결할 때 자격 증명을 검증하는 데 사용하는 보안 자격 증명 집합
- 퍼블릭 키는 Amazon EC2에서 저장되고 프라이빗 키는 사용자가 저장
- 암호 대신 프라이빗 키를 사용하여 인스턴스에 안전하게 액세스

### 테넌시

- AWS 컴퓨팅 테넌시란 EC2 인스턴스가 기본 물리적 하드웨어에 걸쳐 분산되는 방식을 지칭
- 3가지 주요 옵션은 공유 테넌시, 전용 인스턴스, 전용 호스트
- 규정 준수, 라이선스 및 비용 최적화와 관련된 요구 사항에 따라 각기 다른 테넌시를 선택

- **공유 테넌시** - 기본적으로 EC2 인스턴스는 테넌시를 공유. 즉, 여러 AWS 계정에서 동일한 물리적 하드웨어를 공유할 수 있음
- **전용 인스턴스** - 전용 인스턴스는 호스트 하드웨어 수준에서 전용이 아닌 인스턴스와 다른 AWS 계정에 속한 인스턴스로부터 물리적으로 격리되는 EC2 인스턴스
- **전용 호스트** - 고객이 EC2 인스턴스 용량을 완전히 전용으로 사용하는 물리적 서버에서 실행, 사용자가 구성을 제어할 수 있는 격리된 서버가 제공

### 배치 그룹

- Amazon EC2 서비스는 상호 연관된 장애를 최소화하기 위해 기본 하드웨어 전체에 모든 인스턴스를 분산
- 배치 그룹을 사용하면 워크로드 요구를 충족하도록 상호 종속된 인스턴스 집합의 배치 방식을 결정할 수 있음

- **클러스터 배치 그룹** - 낮은 네트워크 지연 시간, 높은 네트워크 처리량 또는 2가지 요건이 모두 충족되면 혜택을 받는 애플리케이션에 권장. 또한 대부분의 네트워크 트래픽이 그룹의 인스턴스 간에 전송되는 경우에도 권장
- **분산형 배치 그룹** - 서로 분리해야 하는 소수의 중요 인스턴스가 포함된 애플리케이션에 권장
- **파티션 배치 그룹** - 대규모 분산 및 복제된 워크로드를 배포할 수 있음, 파티션을 사용하면 여러 구성 요소에서 하드웨어 장애가 동시에 발생하는 상황을 방지할 수 있음

### 사용자 데이터

- EC2 인스턴스를 생성할 때 사용자 데이터를 인스턴스에 전달할 수 있는 옵션이 있으며, 사용자 데이터는 인스턴스 시작 완료를 자동화할 수 있음
- 예를 들어 인스턴스 AM를 패치 및 업데이트하거나, 소프트웨어 라이선스 키를 가져와 설치하거나, 추가 소프트웨어를 설치할 수 있음

### 요금 옵션

- **온디맨드** - 장기 약정 없이 컴퓨팅 용량 비용을 초나 시간 단위로 지불
- **절감형 플랜** - 1년 또는 3년의 특정 기간 동안 확실한 약정을 통해 비용을 절감
- **스팟 인스턴스** - 여유 EC2 호스트 용량을 사용하여 온디맨드 대비 최대 90%의 비용을 절감할 수 있음. 대폭 할인된 요금으로 미사용 EC2 인스턴스를 요청할 수 있으므로, 유동적으로 수행할 수 있는 워크로드에 적합
	- 인터럽트는 현재 최고가에서 요청에 사용 가능한 용량이 현재 없는 경우로, 이벤트 2분 전에 알림을 주며, 2분 후에는 워크로드 실행이 중단됨

# 컨테이너 서비스

- 컨테이너는 한 컴퓨팅 환경에서 다른 컴퓨팅 환경으로 이동할 때 안정적으로 실행되도록 함
- 컨테이너는 코드 및 해당 종속성을 패키징하는 표준화된 단위로, 모든 플랫폼에서 안정적으로 실행되도록 설계 (컨테이너는 자체 독립 환경을 만들기 때문)
- 컨테이너를 사용하면 워크로드를 한 곳에서 다른 곳으로 옮길 수 있음 (예: 개발 환경에서 프로덕션 환경으로 또는 온프레미스 환경에서 클라우드로)

### VM과 컨테이너의 차이

- 컨테이너는 컨테이너가 존재하는 호스트와 동일한 운영 체제와 커널을 공유
- 그러나 가상 머신은 자체적으로 운영 체제를 포함하며, 리소스가 낭비됨

![[../80_Assets/Pasted image 20260707230000.png]]

### Amazon ECS로 컨테이너 관리

- Amazon ECS는 새로운 컨테이너를 가동하는 데 도움이 되는 엔드 투 엔드 컨테이너 오케스트레이션 서비스
- Amazon ECS에서는 컨테이너를 실행하는 머신이 EC2 인스턴스이며, 이 인스턴스에 ECS 에이전트가 설치되어 있고, 컨테이너를 실행하고 관리하도록 구성되어 있음

![[../80_Assets/Pasted image 20260707230442.png]]

- Amazon ECS 컨테이너 인스턴스가 실행되면 △컨테이너 시작 및 중지, △클러스터 상태 확인, △스케일 인 및 스케일 아웃, △클러스터 전체에서 컨테이너 배치 예약, △권한 할당, △가용성 요구 사항 충족 등을 수행할 수 있음

### Amazon EKS에서 Kubernetes 사용

- Kubernetes는 컨테이너화된 워크로드 및 서비스를 관리하기 위한 이동 가능하고 확장 가능한 오픈 소스 플랫폼
- Kubernetes를 이미 사용하고 있다면 Amazon EKS를 사용하여 AWS 클라우드에서 워크로드를 오케스트레이션할 수 있음

# 서버리스

- EC2 인스턴스를 하나도 관리하지 않고 워크로드와 애플리케이션을 배포하고 싶다면 서버리스 컴퓨팅을 사용하면 됨
	- 프로비저닝하거나 관리할 서버가 없습니다.
	- 사용량에 따라 규모가 조정됩니다.
	- 유휴 리소스에 대해서는 비용을 지불하지 않습니다.
	- 가용성 및 내결함성이 내장되어 있습니다.

### AWS Fargate

- AWS Fargate는 컨테이너용 목적별 서버리스 컴퓨팅 엔진
- AWS Fargate가 인프라를 관리하고 규모를 조정하므로 개발자는 가장 잘할 수 있는 애플리케이션 개발 작업에 집중
- Fargate는 Amazon ECS 및 Amazon EKS 아키텍처를 모두 지원

### AWS Lambda

- Lambda를 사용하면 서버를 프로비저닝하거나 관리할 필요 없이 코드를 실행할 수 있음
- 거의 모든 유형의 애플리케이션 또는 백엔드 서비스에 대해 코드를 실행할 수 있음
- Lambda에서 지원하는 언어 중 하나로 소스 코드만 업로드하면 높은 가용성으로 코드를 실행하고 규모를 조정하는 데 필요한 모든 것을 Lambda가 처리

# 클라우드 스토리지

- 블록, 객체, 파일 스토리지는 각기 다른 사용 사례에 적합한 3가지 데이터 스토리지 유형

![[../80_Assets/Pasted image 20260708013851.png]]

- AWS EBS 등의 **블록 스토리지**에서는 데이터가 고정 크기 블록으로 분할. 각 블록에는 고유한 주소가 지정. 짧은 지연 시간 내에 액세스할 수 있어야 하는 부팅 볼륨, 운영 체제, 데이터베이스의 경우에는 대개 블록 스토리지가 적합
- AWS EFS 등의 **파일 스토리지**에서는 대다수 사용자가 잘 알고 있는 계층적 폴더 구조로 데이터를 구성. 대개 여러 시스템 간의 공유 액세스용으로 적합한 파일 스토리지는 표준 파일 프로토콜을 사용
- AWS S3 등의 **객체 스토리지**에서는 고유한 식별자 및 메타데이터를 사용하여 데이터를 개별 단위(객체)로 관리. 대량의 비정형 데이터를 저장하는 데 적합

### 블록 스토리지

- **Amazon EC2 인스턴스 스토어**
	- 이 스토리지는 호스트 컴퓨터에 물리적으로 연결된 디스크에 위치
	- 인스턴스에 블록 수준의 임시 스토리지를 제공
	- 인스턴스 스토어는 버퍼, 캐시, 스크래치 데이터 및 기타 임시 콘텐츠와 같이 자주 변경되는 정보를 임시로 저장하는 데 적합
	- 인스턴스 스토어 볼륨에 저장된 데이터는 연결된 인스턴스의 수명 기간 동안에만 유지
	- 해당 인스턴스를 중지하거나 절전 모드로 전환하거나 종료하면 인스턴스 스토어 볼륨의 모든 데이터가 손실
	- 인스턴스 유형에 따라 사용 가능한 인스턴스 스토어의 크기와 인스턴스 스토어 볼륨에 사용되는 하드웨어 유형이 결정
	- 인스턴스 스토어 볼륨은 인스턴스 사용 비용의 일부로 포함

- **Amazon EBS (Elastic Block Store)**
	- Amazon EC2 인스턴스에 연결할 수 있는 블록 수준 스토리지
	- 외장 드라이브를 랩톱에 연결한다고 생각하면 됨
	- 볼륨과 인스턴스가 동일한 가용 영역에 있어야 함
	- 이 볼륨은 인스턴스에 탑재되므로 데이터가 저장된 위치와 인스턴스 위치 간에 매우 짧은 지연 시간을 제공 (이런 이유로 데이터베이스를 실행하는 데 사용할 수 있음)
	- EC2 인스턴스에서 EBS 볼륨을 분리한 후 동일한 가용 영역 내의 다른 EC2 인스턴스에 연결하여 데이터에 액세스할 수도 있음
	- 사고가 발생하여 컴퓨터가 다운되더라도 외장 드라이브에 데이터가 남아 있음. EBS 볼륨도 마찬가지
	- 데이터의 특정 시점 복사본으로 EBS 스냅샷을 생성할 수 있음 (스냅샷은 S3에 보관)
	- 볼륨에 저장할 수 있는 콘텐츠의 양에 최대 제한이 있음
	- 단일 인스턴스에 여러 EBS 볼륨을 연결할 수 있음
	- 대부분의 EBS 볼륨은 한 번에 1개의 컴퓨터에만 연결할 수 있음 (볼륨 및 인스턴스 유형에 따라 다중 연결을 사용하여 볼륨을 여러 인스턴스에 동시에 탑재할 수 있음)
	- Amazon EBS는 여러 종류의 볼륨 유형을 제공
		- **SSD 지원 볼륨** - I/O 크기가 작은 읽기 및 쓰기 작업을 자주 처리하는 트랜잭션 워크로드에 최적화. 기준 성능 속성은 IOPS (종류: gp2, gp3, io1, io2)
		- **HDD 지원 볼륨** - 주요 성능 속성이 처리량인 대규모 스트리밍 워크로드에 최적화 (종류: st1, sc1)

### 파일 스토리지

- **Amazon EFS (Elastic File System)**
	- 파일을 추가하고 삭제함에 따라 자동으로 확장되고 축소되는 특징이 있어 한 번 설정하고 나면 신경쓰지 않아도 되는 파일 시스템
	- AWS 컴퓨팅 서비스 및 온프레미스 리소스와 함께 사용할 수 있음
	- 여러 개의 컴퓨팅 인스턴스를 동시에 Amazon EFS 파일 시스템에 연결할 수 있음

- **Amazon FSx**
	- 클라우드에서 고성능 파일 시스템을 쉽고 비용 효율적으로 시작, 실행, 규모 조정하는 데 도움이 되는 다양한 기능과 함께 신뢰성, 보안, 확장성을 제공하는 완전관리형 서비스

### 객체 스토리지

- **Amazon S3 (Simple Storage Service)**
	- 컴퓨팅과 연동되지 않는 독립형 스토리지 솔루션
	- 웹의 어느 곳에서나 데이터를 검색할 수 있음
	- 객체 스토리지는 데이터를 평면 구조에 저장함. 객체는 메타데이터와 결합된 파일
	- 이러한 객체를 원하는 만큼 저장할 수 있음
	- Amazon S3에서는 버킷이라는 컨테이너에 객체를 저장
	- 버킷에 객체를 저장할 때 버킷 이름, 키, 버전 ID가 조합되어 객체를 고유하게 식별

| ------------------------------------------------------------------------------- | ----------------------------------------------------- |
| 엄격한 스키마 규칙 및 데이터 품질 적용이 필요합니다.                                                  | 데이터베이스 규모를 수평으로 조정해야 합니다.                             |
| 데이터베이스가 과도한 읽기 및 쓰기 용량을 필요로 하지 않습니다.                                            | 데이터가 기존 스키마에 적합하지 않습니다.                               |
| 최상의 성능을 필요로 하지 않는 관계형 데이터세트의 경우 관계형 데이터베이스 관리 시스템이 자원 소비가 적은 최고의 솔루션이 될 수 있습니다. | 읽기 및 쓰기 속도가 기존 SQL 데이터베이스에서 경제적으로 지원할 수 있는 범위를 초과합니다. |

- 클라우드에서 데이터베이스를 실행할 때 EC2 인스턴스에 데이터베이스를 설치할 수도 있고, Amazon RDS 등의 관리형 데이터베이스 옵션을 선택할 수도 있음

![[../80_Assets/Pasted image 20260708234620.png]]

### Amazon Relational Database Service (RDS)

- Amazon RDS는 기존 데이터베이스 관리의 운영 부담 없이 클라우드에서 관계형 데이터베이스를 생성하고 관리하는 데 사용할 수 있는 관리형 데이터베이스 서비스
- Amazon RDS는 자동으로 데이터베이스 소프트웨어를 패치하고 데이터베이스를 백업
- 프로비저닝, 패치, 스케일링, 복구 같은 인프라 관련 태스크에 집중하는 대신 애플리케이션을 차별화하는데 집중할 수 있음
- Amazon RDS 엔진으로 Amazon Aurora, MySQL, PostgreSQL, MariaDB, Oracle, SQL Server 등 여러 종류를 지원
- DB 인스턴스 아래에는 EC2 인스턴스가 있음. 그러나 이 인스턴스는 Amazon EC2 콘솔 대신 Amazon RDS 콘솔을 통해 관리

- 다중 AZ 배포를 통해 데이터베이스 인스턴스의 가용성 및 내구성을 높일 수 있음. 다중 AZ DB 인스턴스를 프로비저닝하는 경우 RDS에서 다른 AZ의 대기 인스턴스에 데이터를 동기식으로 복제
![[../80_Assets/Pasted image 20260721144638.png]]

- 읽기 전용 복제본을 생성할 수 있음. Amazon에서 자동으로 이러한 복제본을 프라이머리 DB 인스턴스와 동기화함
	- 추가 읽기 용량으로 프라이머리 노드에 대한 부하를 해소
	- 여러 AWS 리전에서 데이터를 애플리케이션에 더 가까이 배치할 수 있음
	- 프라이머리 DB 인스턴스에 장애가 발생하는 경우 재해 복구(DR) 솔루션으로 읽기 전용 복제본을 독립 실행형 인스턴스로 승격할 수 있음
![[../80_Assets/Pasted image 20260721144918.png]]

- Amazon RDS는 AWS Key Management Service(AWS KMS)를 사용하여 저장 데이터 암호화를 제공함
	- AWS KMS는 암호화 키를 생성 및 관리하고 해당 키를 사용하여 데이터를 암호화 및 복호화하는 기능을 제공하는 관리형 서비스

- **데이터베이스 인스턴스**는 특정 컴퓨터 및 스토리지 요구 사항을 제공하는 격리된 데이터베이스 환경
	- 인스턴스 클래스에 따라 RDS DB 인스턴스의 컴퓨팅 및 메모리 용량이 결정
	- Amazon RDS는 표준, 메모리 최적화, 순간 확장 가능 성능 등 3가지 유형의 인스턴스 클래스를 지원

- Amazon RDS는 Amazon EBS 볼륨을 데이터베이스 및 로그 **스토리지**로 사용
	- 스토리지 유형에 따라 각각 성능 특성 및 가격이 다름
	- 범용 SSD, 프로비저닝된 IOPS, 마그네틱 스토리지

- DB 생성 시 오버 프로비저닝하지 않고 필요에 따라 **스토리지를 추가**하도록 스케일링 구성을 사전 설정할 수 있음
- 또한 CPU 또는 메모리 용량을 변경해야 하는 경우 **DB 인스턴스 클래스를 변경**할 수 있음

### Amazon DynamoDB

- DynamoDB는 완전관리형 NoSQL 데이터베이스 서비스로, 원활한 확장성과 함께 빠르고 예측 가능한 성능을 제공
- 데이터 규모와 관계없이 데이터를 저장 및 검색하고, 어떤 수준의 요청 트래픽이라도 처리할 수 있는 데이터베이스 테이블을 생성할 수 있음
- 가동 중지 또는 성능 저하 없이 테이블의 처리량을 스케일 업 또는 스케일 다운 가능
- AWS Management Console을 사용하여 리소스 사용률 및 성능 지표를 모니터링

![[../80_Assets/Pasted image 20260709001839.png]]

- 테이블은 항목의 모음이고 각 항목은 속성의 모음

### 그 외 목적별 데이터베이스

- **Amazon ElastiCache** - 완전관리형 인 메모리 캐싱 솔루션으로, 2개의 오픈 소스 및 인 메모리 캐시 엔진인 Redis와 Memcached를 지원
- **Amazon MemoryDB for Redis** - 내구성이 뛰어나고 Redis와도 호환되는 인 메모리 데이터베이스 서비스로, 초고속 성능을 구현
- **Amazon DocumentDB(MongoDB 호환)** - 문서 데이터베이스, 애플리케이션에서 풍부한 내용의 문서를 저장 및 쿼리하는 데 사용할 수 있는 NoSQL 데이터베이스
- **Amazon Keyspaces(Apache Cassandra용)**
- **Amazon Neptune** - 그래프 데이터베이스, 다양한 관계를 통해 고도로 연결된 데이터에 유용한 옵션
- **Amazon Timestream** - 사물 인터넷(IoT)과 운영 애플리케이션을 위한 빠르고 확장 가능한 서버리스 시계열 데이터베이스 서비스
