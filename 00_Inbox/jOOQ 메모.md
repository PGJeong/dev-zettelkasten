**통계성 데이터**란 수많은 로우(Row)를 읽어서 합계(`SUM`), 평균(`AVG`), 순위(`RANK`), 그룹화(`GROUP BY`) 등을 거쳐 **유의미한 수치나 요약본을 만들어낸 데이터**를 말합니다. (예: 월별 매출 현황, 연령대별 서비스 이용률, 실시간 랭킹 등)

이런 조회에 JPA보다 **jOOQ**가 더 유리한 이유는 다음과 같습니다.

1. "객체"가 필요 없기 때문 (JPA의 오버헤드)

- **JPA**: 데이터를 가져와서 '엔티티 객체'로 만들고, 영속성 컨텍스트에 담아 변경을 추적하는 과정을 거칩니다. 통계는 단순히 숫자나 요약된 정보만 필요한데, 수백만 건의 데이터를 엔티티 객체로 변환하는 것은 메모리 낭비와 성능 저하를 초래합니다.
- **jOOQ**: SQL을 실행한 결과를 바로 DTO(단순 데이터 전송 객체)나 Map 형태로 매핑하므로 군더더기 없이 가볍고 빠릅니다.

2. 복잡한 SQL 기능(Window Function 등) 활용

통계 쿼리에는 단순 Join 외에 고급 기능이 많이 쓰입니다.

- **윈도우 함수**: `RANK()`, `ROW_NUMBER()`, 누적 합계 등.
- **Pivot/Unpivot**: 가로 데이터를 세로로, 세로를 가로로 변환.
- **CTE (WITH 절)**: 복잡한 서브쿼리를 가독성 있게 구조화.
- **jOOQ**는 이 모든 최신 표준 SQL 문법을 자바 코드로 완벽하게 지원합니다. 반면 JPA(JPQL)는 이런 기능을 지원하지 않거나, 구현하더라도 문법이 매우 제한적입니다.

3. SQL 가독성과 최적화 (SQL 중심 사고)

- 통계는 한 끝 차이로 실행 속도가 수십 분에서 수 초로 갈립니다. 따라서 인덱스 힌트를 쓰거나 특정 순서로 실행되도록 SQL을 직접 튜닝해야 할 때가 많습니다.
- JPA는 "객체 그래프"를 따라가며 쿼리를 생성하기 때문에 SQL을 미세하게 제어하기 어렵지만, **jOOQ는 내가 쓴 자바 코드가 곧 SQL**이므로 쿼리 튜닝이 직관적입니다.
